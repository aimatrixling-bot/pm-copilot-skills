# 用户画像

<SUBAGENT-STOP>
如果是子代理派发执行此 Skill，直接按下方指令执行，不重新加载入口 Skill。
</SUBAGENT-STOP>

创建基于数据/研究的用户画像——从"我觉得用户是这样的"到"基于证据的用户画像"。

**核心原则**：用户画像是决策工具，不是装饰品。没有痛点的画像只是人口统计报告。

## Iron Law（铁律）

| 铁律 | 违反后果 |
| --- | --- |
| 画像必须基于数据，不可纯虚构 | 立即停止——标注来源类型（访谈/问卷/数据分析/假设），纯假设标注 [假设] |
| 每个画像必须包含痛点 | 立即停止——补充 ≥ 1 个痛点后继续 |
| 画像数量限制 3-7 个 | 立即停止——< 3 个补充缺失群体，> 7 个合并相似画像 |

## 反合理化

| 你可能在想的 | 真相 |
| --- | --- |
| "我很了解用户，不需要画像" | 你了解的是典型用户，不是全部用户。画像暴露盲区 |
| "用户都差不多，做一个就行" | "都差不多" = 你没发现差异 |
| "画像写完就完了" | 画像是活文档，需要随用户研究持续更新 |
| "直接用竞品的画像" | 竞品的用户 ≠ 你的用户 |
| "人口统计够了" | 年龄/收入/地区 ≠ 用户需求。行为和痛点才是决策依据 |

## Entry Mode

### Guided（引导模式）— 6-8 步

1. **目标产品/功能** — 为哪个产品/功能服务？
2. **已有认知** — 有用户研究/有一些假设/几乎没有
3. **人口统计** — 年龄、职业、地域、收入（不确定标注 [假设]）
4. **行为特征** — 使用习惯、信息获取、决策模式
5. **痛点与需求** — 用户想完成什么但做不到？5 Whys 追溯根因
6. **使用场景** — 典型的一天怎么用这个产品？
7. **画像数量确认** — 建议 X 个画像 → 用户确认
8. **产出确认** — 完整画像 → 确认交付

### Quick（快速模式）— ≤ 2 个问题

**问题 1**: "产品/功能是什么？核心用户大概是什么群体？"
**问题 2**（可选）: "你知道他们最大的痛点吗？"

然后：自动推断 3-5 个画像 → 每个包含名称+描述+人口统计+行为+痛点+场景 → 标注 [假设] → 识别 Primary Persona

## 执行流程

```
触发 pm-persona
    ├── 1. 模式判断（Guided/Quick）
    ├── 2. 读取上下文（Glob 搜索 PRD/Roadmap）
    ├── 3. 按模式执行
    ├── 4. 构建画像（3-7 个，含名称+描述+人口统计+行为+痛点+场景）
    ├── 5. Iron Law 检查
    │     ├── 来源标注？
    │     ├── 每个有痛点？
    │     └── 数量 3-7？
    └── 6. 交付 + 建议下一步（→ pm-prd 或 → pm-experiment）
```

## Hard Bans（硬禁令）

以下模式在所有保真度级别均禁止出现。检测到时立即修正。

| # | 禁止模式 | 修正动作 |
| --- | --------- | -------- |
| HB-1 | 画像只有人口统计没有行为和痛点 | 补充行为特征和 >=1 个具体痛点 |
| HB-2 | 画像无数据来源（纯虚构人物） | 标注画像依据（访谈/问卷/数据推断），无法验证的标注 [假设] |
| HB-3 | Primary Persona 未明确选择 | 明确标注 Primary Persona 并说明选择理由 |
| HB-4 | 痛点无场景描述 | 每个痛点补充具体的使用场景和触发条件 |
| HB-5 | 多个画像之间无差异度 | 明确每个画像的独特性，避免画像间特征重叠 > 60% |

## 交付前检查

- [ ] 画像数量在 3-7 个范围
- [ ] 每个画像有名称和一句话描述
- [ ] 包含人口统计（≥ 3 维度）和行为特征（≥ 2 个）
- [ ] 每个画像有 ≥ 1 个痛点和 ≥ 1 个使用场景
- [ ] 已识别 Primary Persona
- [ ] 所有推断标注 [假设]，来源类型已标注
- [ ] 相似画像已合并
- [ ] 画像每个特征有数据来源标注（访谈/问卷/数据推断）
- [ ] 痛点和场景描述区分了事实观察和推断

## Source Attribution（来源标注规范）

### 画像特征来源分类

| 特征类型 | 来源要求 | 标注方式 |
|---------|---------|---------|
| **人口统计** | 实际数据或合理推断 | `[来源: 用户数据/行业报告]` |
| **行为特征** | 访谈/问卷/行为数据 | `[来源: 用户访谈 N 人/行为数据]` |
| **痛点** | 至少 1 个一手证据 | `[来源: 访谈记录/客服工单/用户反馈]` |
| **目标/动机** | 用户原话或行为推断 | `[来源: 用户原话/推断: 行为X→目标Y]` |
| **假设特征** | 无直接证据，AI 推断 | `[假设]` |

### 检查规则

- Primary Persona 的每个痛点必须有至少 1 个一手证据支撑
- 纯推断的特征标注 [假设]，且数量不超过总特征的 30%
- 画像间相同特征需标注来源一致性

## 画像结构（每个画像必须包含）

| 维度 | 内容 | 示例 |
| --- | --- | --- |
| 名称 | 有辨识度的名字 | "忙碌的运营经理 小王" |
| 一句话描述 | 谁的核心需求是什么 | "需要快速生成报表的运营经理" |
| 人口统计 | 年龄/职业/地域/收入 | 28-35岁，一二线城市 |
| 行为特征 | 使用习惯/信息获取/决策模式 | 习惯用移动端，爱看短视频 |
| 痛点 | ≥1 个具体痛点 | 手动汇总数据太耗时 |
| 使用场景 | 典型使用的一天 | 每周一早上做上周数据汇报 |
| 来源标注 | 访谈/问卷/数据分析/[假设] | 基于用户访谈 |

|------|------|

1. [特征1]
