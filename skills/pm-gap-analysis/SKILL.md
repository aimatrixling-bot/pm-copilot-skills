# 战略偏差诊断

<SUBAGENT-STOP>
如果是子代理派发执行此 Skill，直接按下方指令执行，不重新加载入口 Skill。
</SUBAGENT-STOP>

诊断产品战略、实际构建、用户真实需求三者之间的偏差——不是"做得好不好"，而是"方向对不对"。

**核心价值**：团队在做产品时，战略意图会随时间漂移。这个 Skill 帮你在关键节点做一次"方向校准"，发现那些不知不觉偏离的决策。

**置信度原则**：AI 基于行业知识和最佳实践主动推断，不保守回避。有用户数据时提高置信度；没有数据时基于专业判断给出推断，并明确标注置信等级。标注置信度是为了帮 PM 决定"哪些 gap 需要优先用数据验证"，不是作为"不说"的借口。

## Iron Law（铁律）

| 铁律 | 违反后果 |
| --- | --- |
| 每个 gap 必须标注置信度和数据来源 | 立即停止——补全标注后继续 |
| 不回避判断。缺乏用户数据 ≠ 无法分析 | 缺乏数据时给出专业推断 + 明确标注"推断" |
| gap 必须导向行动。发现问题不是目的，改变方向才是 | 每个 gap 必须附带建议行动 |
| 不制造虚假安心。即使分析结果"整体对齐"，也必须指出潜在风险 | 补充 ≥ 2 个潜在风险点 |

## 反合理化

| 你可能在想的 | 真相 |
| --- | --- |
| "我们的战略很清晰，不需要校准" | 清晰的战略 ≠ 执行没有漂移。最清晰的方向也值得定期检查 |
| "用户反馈很好，说明方向对" | 满意度 ≠ 方向正确。用户可能满意一个偶然做对的功能，而核心价值仍然偏离 |
| "这是产品review，用 pm-critique 就行" | critique 评"做得好不好"，gap-analysis 评"方向对不对"。视角不同 |
| "没有用户数据，分析没意义" | 行业知识 + 竞品认知 + 最佳实践足以做出有价值的推断。标注置信度即可 |
| "gap 分析太抽象，不如直接改功能" | 不诊断就改功能 = 盲目迭代。先确认方向，再决定改什么 |

## Step 0: 上下文先行

在问用户任何问题之前，先搜索项目已有信息：

1. **搜索项目文档** — Glob 搜索以下目录和文件：
   - `docs/` 下的战略文档、PRD、roadmap、discovery 产出
   - 项目根目录的 `CLAUDE.md`、`README.md`
   - `research/` 下的竞品分析、行业研究
2. **扫描对话历史** — 提取用户在本次会话中已提及的战略目标、用户定义、功能范围
3. **提取已有信息** — 按三类整理：战略陈述（说的）、已构建/计划构建（做的）、用户相关信息（用户要的）

然后用以下格式向用户确认：
> "我找到了以下项目信息：[信息清单]。我会基于这些信息进行分析，如果有过时或不准确的请告诉我。"

**信息不足时的处理**：
- 如果只找到战略文档，无构建信息 → 基于"说的"推断"应该做什么"，对比"实际应该做什么"
- 如果只找到构建记录，无战略文档 → 基于构建内容反推隐含战略，然后分析这个隐含战略是否合理
- 如果两类都缺 → 要求用户提供最低必要信息（一句话战略 + 当前在做的事）

## 分析框架：五维偏差诊断

对以下五个维度逐一分析，每个维度输出 Gap 评估：

### 维度 1：目标用户对齐

| 检查点 | 说明 |
| --- | --- |
| 战略定义的用户是谁 | 从文档中提取目标用户画像 |
| 实际功能服务的是谁 | 分析功能列表/PRD 面向的用户场景 |
| 是否存在"战略用户"和"功能用户"不一致 | 比如战略说"服务中小企业"，功能却只适合大企业 |

**AI 推断能力**：基于行业知识和产品类型，推断目标用户群体的典型需求和行为模式。即使没有用户调研数据，也可以基于：
- 同类产品的用户特征
- 行业最佳实践中的用户分层
- 产品功能设计隐含的用户假设

### 维度 2：价值主张落地

| 检查点 | 说明 |
| --- | --- |
| 战略声称的差异化价值是什么 | 提取核心价值主张 |
| 已构建/计划的功能是否支撑这个价值 | 逐个功能检查与价值主张的关联 |
| 是否有功能与价值主张无关（scope 蔓延） | 识别不在价值主张范围内却消耗资源的功能 |

**AI 推断能力**：基于同类产品的价值定位模式，判断价值主张是否常见/差异化不足，以及功能组合是否足以兑现价值承诺。

### 维度 3：优先级偏离

| 检查点 | 说明 |
| --- | --- |
| Roadmap/Backlog 中的 P0 是什么 | 提取最高优先级事项 |
| 实际投入（时间/资源）最多的是什么 | 从构建记录或用户描述推断 |
| P0 和实际投入是否一致 | 资源错配 = 隐含的优先级偏离 |

**AI 推断能力**：基于产品阶段（探索期/成长期/成熟期）判断优先级是否合理。比如探索期大量投入基础设施而非用户验证，就是典型的优先级偏离。

### 维度 4：用户假设验证

| 检查点 | 说明 |
| --- | --- |
| 产品隐含了哪些用户假设 | 从功能设计反推假设（如"用户需要实时响应"） |
| 这些假设是否有数据支撑 | 区分"已验证"和"未验证" |
| 未验证假设中哪些风险最高 | 影响 core value prop 的假设 = 高风险 |

**AI 推断能力**：这是 AI 最能发挥推断能力的维度。基于行业经验和用户行为规律，评估假设的合理性。例如：
- 医疗行业用户假设需要考虑合规性和信任门槛
- To B 产品假设需要考虑决策链条和采购流程
- 消费级产品假设需要考虑注意力和使用频率

### 维度 5：竞品差异化

| 检查点 | 说明 |
| --- | --- |
| 战略定位与竞品的差异在哪 | 从竞品分析文档或推断中提取 |
| 实际功能是否体现了这个差异 | 功能层面的差异化验证 |
| 差异化是否被竞品轻易复制 | 差异的可持续性 |

**AI 推断能力**：基于行业竞品格局和产品特性，判断差异化是否真实、是否可持续。

## 置信度分级

每个 gap 判断使用三级置信度：

| 等级 | 标签 | 含义 | 数据基础 |
| --- | --- | --- | --- |
| 高 | `[高] 基于文档` | 文档中存在明确证据 | 战略文档、PRD、功能列表中的直接矛盾 |
| 中 | `[中] 基于推断` | 无直接数据，但基于行业知识和最佳实践的合理推断 | 同类产品模式、行业惯例、产品逻辑推演 |
| 低 | `[低] 需要验证` | 推断存在不确定性，需要用户数据确认 | 用户行为假设、付费意愿、需求优先级 |

**核心原则**：低置信度 ≠ 不说。低置信度的 gap 反而是最需要标注的——它们正是"需要用数据验证"的候选项。

## 执行流程

```
触发 pm-gap-analysis
    ├── 1. 上下文收集
    │     ├── Glob 搜索项目文档
    │     ├── 提取三类信息：说的 / 做的 / 用户相关的
    │     └── 向用户确认信息完整性
    ├── 2. 五维偏差诊断
    │     ├── 逐维度分析（见上方框架）
    │     ├── 每个维度输出：Gap 描述 + 置信度 + 建议行动
    │     └── 主动推断，不回避判断
    ├── 3. Iron Law 检查
    │     ├── 每个gap有置信度标注？
    │     ├── 缺乏数据时给出了推断？
    │     ├── 每个gap有建议行动？
    │     └── 整体对齐时也指出了风险？
    ├── 4. 汇总与优先排序
    │     ├── 按影响程度排序所有 gap
    │     ├── 标注"立即行动" vs "下次迭代验证"
    │     └── 指出最需要用户数据验证的 1-2 个 gap
    └── 5. 交付 + 建议下一步
          ├── 需要用户调研？ → pm-persona / pm-experiment
          ├── 需要重新定位？ → pm-problem-frame
          └── 需要调整优先级？ → pm-rice / pm-roadmap
```

## 执行模式

| 模式 | 触发信号 | 行为 |
| --- | --- | --- |
| **Full Scan**（默认） | 无特定聚焦信号 | 五维度全扫，输出完整报告 |
| **Focused** | 用户指定维度或方向 | 只分析指定维度，输出更快 |
| **Quick Pulse** | "快速检查"、"大概看看" | 五维度各 1-2 句话判断 + 整体信号灯（绿/黄/红） |
