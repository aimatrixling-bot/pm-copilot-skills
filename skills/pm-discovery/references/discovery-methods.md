# Discovery 方法论详解

> 产品发现的核心方法和验证技巧。

---

## 1. 5 Whys 根因分析

从表面问题追溯到根本原因：

```
表面问题: [用户看到的症状]
    ↓ 为什么？
第 1 层: [直接原因]
    ↓ 为什么？
第 2 层: [系统性原因]
    ↓ 为什么？
第 3 层: [流程缺陷]
    ↓ 为什么？
第 4 层: [组织/文化原因]
    ↓ 为什么？
根本原因: [可行动的根因]
```

### 5 Whys 规则

| 规则 | 说明 |
|------|------|
| 不归咎个人 | "因为张三犯错了" → "因为流程允许手动犯错" |
| 答案必须具体 | "因为设计有问题" → "因为没有自动化测试" |
| 至少追问 5 层 | 前 3 层通常停留在技术面，需要触及流程和组织 |
| 根因必须可行动 | 如果根因无法行动，说明没到真正的根因 |

### 验证方法

| 方法 | 适用场景 |
|------|----------|
| 用户访谈（5-8 人） | 验证问题存在性和严重性 |
| 数据验证 | 用现有数据量化问题规模 |
| 竞品对比 | 看竞品是否已解决类似问题 |
| 快速原型测试 | 用低保真原型测试方案可行性 |

## 2. 门控标准详解

### Phase 1 门控：问题定义

| 检查项 | 通过标准 |
|--------|----------|
| 数据支撑 | 有 ≥ 1 个数据源量化问题 |
| 受影响用户 | 已量化用户群规模 |
| 根因分析 | 已区分表面症状和根本原因 |
| 非解决方案 | 问题描述不包含解决方案 |

### Phase 2 门控：用户画像

| 检查项 | 通过标准 |
|--------|----------|
| 画像数量 | 3-7 个画像 |
| 痛点 | 每个画像 ≥ 1 个痛点 |
| Primary 已识别 | 已选出 Primary Persona |
| 痛点有证据 | 痛点标注了来源（访谈/数据/假设） |

### Phase 3 门控：竞品分析

| 检查项 | 通过标准 |
|--------|----------|
| 差异化结论 | 有明确的差异化方向 |
| 来源标注 | 每个结论有出处 |
| 竞品数量 | ≥ 3 个直接竞品 + ≥ 2 个间接竞品 |
| 差异化空间 | 已识别可利用的空白地带 |

### Phase 4 门控：可行性判断

| 检查项 | 通过标准 |
|--------|----------|
| 决策有依据 | 基于 Phase 1-3 的产出 |
| 方案数量 | ≥ 2 个方案被评估 |
| 行动项 | 有具体行动 + 负责人 + 时间 |
| 决策明确 | Go / No-Go / 暂缓 三选一 |

## 3. Discovery 常见模式

### 双钻模型

```
    发散          收敛        发散          收敛
  ┌─────┐     ┌─────┐    ┌─────┐     ┌─────┐
  │问题  │ ──→ │定义  │ ──→ │方案  │ ──→ │交付  │
  │探索  │     │问题  │    │探索  │     │一个  │
  └─────┘     └─────┘    └─────┘     └─────┘
  Phase 1      Phase 1     Phase 3-4    Phase 4
  + Phase 2    门控        部分         Go/No-Go
```

### 时间盒建议

| 阶段 | 建议时间 | 最多 |
|------|----------|------|
| Phase 1 问题定义 | 2-3 天 | 5 天 |
| Phase 2 用户画像 | 2-3 天 | 5 天 |
| Phase 3 竞品分析 | 3-5 天 | 7 天 |
| Phase 4 可行性判断 | 1-2 天 | 3 天 |
| **总计** | **8-13 天** | **20 天** |

超过 20 天的 Discovery = 分析瘫痪。立即产出当前最佳判断。
