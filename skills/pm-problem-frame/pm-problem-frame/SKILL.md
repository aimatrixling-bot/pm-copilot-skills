# 问题定义

<SUBAGENT-STOP>
如果是子代理派发执行此 Skill，直接按下方指令执行，不重新加载入口 Skill。
</SUBAGENT-STOP>

引导 PM 精准定义产品问题——在跳入解决方案之前。

**核心原则**：正确定义问题 = 解决了一半。问题定义错误是 PM 最大的浪费。

## Iron Law（铁律）

| 铁律 | 违反后果 |
| --- | --- |
| 没有用户的问题不是问题 | 立即停止——明确受影响的用户群体后继续 |
| 没有证据的问题只是假设 | 立即停止——标注 [假设] 并提供验证方法后继续 |
| 混淆问题和方案的定义无效 | 立即停止——剥离方案描述，只保留问题陈述 |

## 反合理化

| 你可能在想的 | 真相 |
| --- | --- |
| "问题很明显，不需要定义" | 明显的问题 ≠ 正确定义的问题 |
| "用户说了痛点" | 痛点 ≠ 问题，需要 5 Whys 挖到根因 |
| "先把方案想好再定义问题" | 先有方案再找问题 = 合理化，不是产品发现 |
| "竞品没解决这个问题" | 竞品没解决 ≠ 用户不需要，可能是机会 |
| "这个数据支撑了问题" | 相关性 ≠ 因果性，一个数据点不构成证据 |

## Step 0: 上下文先行

在问用户任何问题之前，先搜索项目已有信息：

1. **搜索项目文档** — Glob 搜索 `docs/` 目录下的相关文件（problem-statement、personas、PRD、roadmap 等）
2. **扫描对话历史** — 提取用户在本次会话中已提及的关键信息（用户群体、痛点、场景、约束）
3. **提取已有信息** — 列出已知信息清单，标注来源（文档/对话/推断）

然后用以下格式向用户确认：
> "我找到了以下相关信息：[已有信息清单]。我会基于这些信息继续，如果有过时的请告诉我。"

如果已有信息足够丰富，直接跳过探索式对话，进入精准追问或直接产出。

## 自适应流程

收到用户输入后，先执行信息评估：

| 评估维度 | 高密度信号 | 低密度信号 |
| --- | --- | --- |
| 问题/产品描述 | 具体场景 + 用户群体 + 痛点 | 仅一句"帮我定义问题" |
| 上下文丰富度 | 项目已有相关文档、之前讨论过 | 全新话题，无历史 |
| 用户明确度 | 带具体数据/竞品名/功能列表 | 模糊想法 |

**判断规则**：
- 3 个维度都是高密度 → **直接产出**（不提问，所有推断标注 [假设]）
- 混合 → **精准追问**（只问缺失的关键维度，可一次多问有逻辑关联的问题）
- 3 个维度都是低密度 → **探索式对话**（分组提问，最多 3 轮）

**关键**：即使探索式对话，也必须先完成"上下文先行"步骤，再决定问什么。不要问用户已经回答过的问题。

### 关键决策点：严重程度评估

当需要评估问题严重程度时，使用 AskUserQuestion：

> **问题**: "这个问题的严重程度属于哪种？"
>
> | 选项 | 说明 |
> | --- | --- |
> | 阻塞性 | 用户完全无法完成目标 |
> | 严重不便 | 有 workaround 但体验差 |
> | 轻微不便 | 影响体验但不阻碍核心流程 |
> | 体验层面 | 视觉/交互优化 |

仅在信息密度不足以直接推断时使用。能从上下文推断的，直接标注 [假设]。

## 提问纪律

- 允许一条消息中问多个问题，但必须**按逻辑分组**（如：关于用户的 3 个问题放一起，关于场景的 2 个问题放一起）
- 优先问"最可能改变方向"的问题（如用户群体、核心问题），而非"最方便填充模板"的问题（如频率、严重程度）
- 能从上下文推断的信息，不问。标注为 [假设] 并请用户确认
- 每轮追问后，复述你理解的内容，让用户确认或纠正
- 避免超过 5 个问题的追问轰炸——如果需要更多信息，考虑分 2 轮

### 5 Whys 根因分析

**非必做**。只在根因不明确时执行 5 Whys。如果用户已明确说明根因，直接记录。

执行时连续追问（≤ 5 次），直到触达根因。简化版 ≤ 3 层也可接受。

## 执行流程

```
触发 pm-problem-frame
    ├── 0. 上下文先行
    │     ├── 搜索项目已有问题定义文档
    │     ├── 扫描对话历史中已提及的关键信息
    │     └── 向用户确认已有信息
    ├── 1. 信息密度评估
    │     ├── 高密度 → 直接产出（标注 [假设]）
    │     ├── 混合 → 精准追问（只问缺失维度）
    │     └── 低密度 → 探索式对话（分组提问，≤ 3 轮）
    ├── 2. 核心分析
    │     ├── 问题陈述（检查是否混入方案 → 剥离）
    │     ├── 受影响用户（群体 + 规模）
    │     ├── 频率与严重程度（需决策时用 AskUserQuestion）
    │     ├── 根因分析（仅在根因不明时执行 5 Whys）
    │     ├── 现有方案与方案空白
    │     └── 证据评估（强/中/弱）
    ├── 3. Iron Law 检查
    │     ├── 受影响用户是否明确？
    │     ├── 证据是否标注？
    │     └── 问题陈述是否包含方案？（有 → 剥离）
    └── 4. 交付 + 建议下一步（→ pm-rice 或 pm-prd）
```

## Visual Preview（视觉化预览）

### 原则：先看再做

在输出正式内容前，对涉及视觉判断的环节提供预览。用户确认方向后再执行。

### 检测机制

执行 Visual Preview 前检查：
1. Superpowers 插件是否已安装（检查 `~/.claude/plugins/cache/claude-plugins-official/superpowers/` 是否存在）
2. Visual Companion server 脚本是否存在（`skills/brainstorming/scripts/start-server.sh`）
3. 当前环境是否支持后台 server（Windows 需 `run_in_background: true`）

**可用** → 启动 Visual Companion，在浏览器中展示方案
**不可用** → Fallback 到文字输出，不影响主流程

### pm-problem-frame 的 Visual Preview 场景

| 环节 | 展示内容 | 不可用时 |
| --- | --- | --- |
| 5 Whys 根因链 | 因果链可视化（从表面现象到根因的路径图） | 用 Markdown 箭头列表 |
| 问题评级 | 严重程度 × 频率 × 规模的三维评分可视化 | 用表格 |

### 注意事项

- Visual Preview 是**辅助确认工具**，不是必须环节。不可用时绝不阻塞主流程
- 每次只展示一个决策点的内容，不要在一个页面堆砌所有信息
- 预览内容保持简洁，不做像素级设计

## 交付规范

1. **对话中直接输出** — Markdown 格式，不包裹在代码围栏中
2. **标注系统** — 所有推断用 [假设]，自动填充用 [默认]，待讨论用 [待确认]，需数据验证用 [待验证]
3. **文件保存**（可选）— 如果项目有 `docs/` 目录，询问用户是否保存为文件。没有 docs/ 目录则不问，直接在对话中输出
4. **下一步建议** — 基于当前 Main Loop 阶段，推荐 1-2 个最相关的后续 Skill

## Hard Bans（硬禁令）

以下模式在所有保真度级别均禁止出现。检测到时立即修正。

| # | 禁止模式 | 修正动作 |
| --- | --------- | -------- |
| HB-1 | 问题陈述中包含解决方案描述 | 剥离方案，只保留问题陈述 |
| HB-2 | 受影响用户只有"用户"无具体群体 | 明确受影响的用户群体和规模 |
| HB-3 | 相关性表述暗示因果关系 | 改为"与...相关"并标注 [相关性非因果] |
| HB-4 | 证据只有 1 个来源支撑核心判断 | 核心判断至少 2 个独立来源，或标注 [单一来源-置信度低] |
| HB-5 | 频率/严重程度未量化 | 补充具体频率（每天/每周等）和严重程度级别 |

## 交付前检查

- [ ] 问题陈述不包含解决方案描述
- [ ] 受影响用户已明确（群体 + 规模估算）
- [ ] 频率和严重程度已量化
- [ ] 根因分析已完成（仅在根因不明时执行 5 Whys）
- [ ] 证据强度已评估（强/中/弱）
- [ ] 弱证据已标注 [待验证] 并建议验证方法
- [ ] 问题评级已计算
- [ ] Problem Statement 使用标准格式
- [ ] 所有证据已标注来源和强度
- [ ] 推断性结论已区分事实与推断

## Source Attribution（来源标注规范）

### 证据分类

| 类型 | 标注方式 | 最低要求 |
|------|---------|---------|
| **强证据** | >=2 个独立来源交叉验证 | 用户访谈 + 数据验证 |
| **中证据** | 1 个来源 + 逻辑推理 | 单次调研或历史数据 |
| **弱证据** | 单一来源或 AI 推断 | 标注 [待验证] 并提供验证方法 |

### 检查规则

- Problem Statement 中每个定量断言必须有来源
- 证据表格中每个条目有来源 + 强度评级
- 核心因果判断至少"中"强度证据支撑
- 弱证据不可单独支撑问题评级

```markdown

> **定义者**: [用户名] (AI-assisted)
> **日期**: YYYY-MM-DD
> **证据强度**: [强/中/弱]

## Problem Statement
[标准格式："[用户群体] 在 [场景] 中遇到 [问题]，导致 [影响]。这个问题 [频率] 发生，严重程度为 [级别]。根因是 [根因]。现有方案 [workaround]，但存在 [方案空白]。"]

## 受影响用户
- **群体**: [用户群体]
- **规模**: [估算数量]
- **频率**: [每天/每周/每月/偶尔]
- **严重程度**: [阻塞性/严重不便/轻微不便/美观]

## 根因分析
1. 为什么会发生？→ [原因 1]
2. 为什么会导致 [原因 1]？→ [原因 2]
3. ... → [根因]

## 证据
| 证据 | 来源 | 强度 |
|------|------|------|
| [证据 1] | [来源] | [强/中/弱] |

## 现有方案
[用户目前的 workaround]

## 问题评级
**[高/中/低]** — 频率 × 严重程度 × 规模

## 标注
**[假设]** [推断的内容]
**[待确认]** [需要用户确认的内容]
**[待验证]** [需要数据验证的内容]
```

## 与 PM Copilot Agent 的协同

本 Skill 在 PM Copilot Agent 内运行时，遵循 Agent 的执行协议：

1. **澄清轮次**：最多 2 轮追问。如果 2 轮后信息仍不足，用当前最佳判断产出，标注 [待确认]
2. **路由感知**：检查用户输入是否匹配更高优先级的 Skill（如用户说"写 PRD"但实际需要先定义问题 → 先推荐 pm-problem-frame）
3. **Main Loop 衔接**：产出完成后，明确建议 Main Loop 中的下一步 Skill

在 Claude Code 中直接调用时（不经过 Agent），不受 2 轮限制，可按自适应流程正常展开。
