# 问题定义

<SUBAGENT-STOP>
如果是子代理派发执行此 Skill，直接按下方指令执行，不重新加载入口 Skill。
</SUBAGENT-STOP>

引导 PM 精准定义产品问题——在跳入解决方案之前。

**核心原则**：正确定义问题 = 解决了一半。问题定义错误是 PM 最大的浪费。

## Iron Law（铁律）

| 铁律 | 违反后果 |
| --- | --- |
| 没有用户的问题不是问题 | 立即停止——明确受影响的用户群体后继续 |
| 没有证据的问题只是假设 | 立即停止——标注 [假设] 并提供验证方法后继续 |
| 混淆问题和方案的定义无效 | 立即停止——剥离方案描述，只保留问题陈述 |

## 反合理化

| 你可能在想的 | 真相 |
| --- | --- |
| "问题很明显，不需要定义" | 明显的问题 ≠ 正确定义的问题 |
| "用户说了痛点" | 痛点 ≠ 问题，需要 5 Whys 挖到根因 |
| "先把方案想好再定义问题" | 先有方案再找问题 = 合理化，不是产品发现 |
| "竞品没解决这个问题" | 竞品没解决 ≠ 用户不需要，可能是机会 |
| "这个数据支撑了问题" | 相关性 ≠ 因果性，一个数据点不构成证据 |

## Entry Mode

### Guided（引导模式）— 预计 5-8 个问题

**Step 1: 问题陈述**
问："你观察到的现象是什么？请描述你看到/听到的事情（不要包含解决方案）。"
→ 检查是否混入方案 → 如果混入，剥离方案，保留现象

**Step 2: 受影响用户**
问："谁遇到了这个问题？大约多少人？"
→ 记录 Target Users、规模估算

**Step 3: 频率与严重程度**
问："这个问题多久发生一次？对用户的影响有多严重？"
频率：A. 每天 B. 每周 C. 每月 D. 偶尔
严重程度：A. 阻塞性 B. 严重不便 C. 轻微不便 D. 美观/体验
→ 记录 Frequency、Severity

**Step 4: 5 Whys（根因分析）**
基于问题陈述执行连续追问（≤ 5 次），直到触达根因。
→ 记录 Root Cause

**Step 5: 现有方案**
问："用户目前怎么解决或绕过这个问题？"
→ 记录 Workaround，识别方案空白

**Step 6: 证据检查**
问："你有什么证据证明这个问题真实存在？"
→ 评估证据强度：强（多源数据）/ 中（单一来源）/ 弱（仅直觉）
→ 弱证据标注 [待验证] 并建议验证方法

**Step 7: 问题评级**
频率 × 严重程度 × 用户规模 → 高/中/低

**Step 8: 问题陈述精炼**
自动生成标准格式：
"[用户群体] 在 [场景] 中遇到 [问题]，导致 [影响]。这个问题 [频率] 发生，严重程度为 [级别]。根因是 [根因]。现有方案 [workaround]，但存在 [方案空白]。"

### Quick（快速模式）— ≤ 2 个问题后直接产出

**问题 1**: "你要解决什么问题？谁受影响？"
**问题 2**（可选）: "你知道根因是什么吗？"

然后：
- 自动推断频率、严重程度、现有方案 → 标注 [假设]
- 简化版 5 Whys（≤ 3 层）
- 产出问题定义文档，包含 [待确认] 和 [待验证] 标记

### Expert（专家模式）— 直接产出

基于对话上下文中的信息，直接产出完整的问题定义文档。所有推断标注 [假设]，所有缺失信息标注 [待确认]。

## 执行流程

```
触发 pm-problem-frame
    ├── 1. 模式判断（Guided/Quick/Expert）
    ├── 2. 读取上下文（搜索项目已有问题定义文档）
    ├── 3. 按模式执行
    ├── 4. Iron Law 检查
    │     ├── 受影响用户是否明确？
    │     ├── 证据是否标注？
    │     └── 问题陈述是否包含方案？（有 → 剥离）
    └── 5. 交付 + 建议下一步（→ pm-rice 或 pm-prd）
```

## 交付前检查

- [ ] 问题陈述不包含解决方案描述
- [ ] 受影响用户已明确（群体 + 规模估算）
- [ ] 频率和严重程度已量化
- [ ] 5 Whys 根因分析已完成
- [ ] 证据强度已评估（强/中/弱）
- [ ] 弱证据已标注 [待验证] 并建议验证方法
- [ ] 问题评级已计算
- [ ] Problem Statement 使用标准格式
