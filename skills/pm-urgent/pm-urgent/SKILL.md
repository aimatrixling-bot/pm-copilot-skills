# 紧急问题响应

紧急不是跳过思考的理由，是加速思考的信号。紧急模式下省略的是探索，不是判断。核心风险：紧急=跳过方法论=技术债+决策债。

## Scope Gate

- **IN**: 紧急 PRD、快速决策、危机响应、hotfix 需求文档、紧急竞品扫描、快速澄清问题
- **OUT**: 长期战略规划、完整产品发现流程、深度技术评审（→对应标准 Skill）

## 保真度分级

| 保真度 | 检查范围 |
| --- | --- |
| Draft（紧急默认） | 4 条 Iron Law 基础检查，跳过交叉检查 |
| Review | + 风险评估 + 回滚方案 + 紧急决策记录 |
| Release | 不适用紧急模式（紧急上线仍需走完整 Review 流程） |

## Iron Law（铁律）

| 铁律 | 违反后果 |
| --- | --- |
| 每个问题必须附"为什么问这个" | 去掉——没有理由的问题浪费注意力 |
| 不超过 7 个问题 | 超过 = 认知过载，用户不会回答 |
| 提供快速回答模板 | 没模板 = 用户要自己组织语言，降低回复率 |

## 反合理化

| 你可能在想的 | 真相 |
| --- | --- |
| "问得越多越好" | 7 个以上 = 用户直接放弃 |
| "问题要全面覆盖" | 时间紧 = 只问阻塞进度的关键问题 |
| "先给答案再问问题" | 没搞清楚问题就给答案 = 浪费双方时间 |

## Entry Mode

紧急模式只有 Quick/Expert，无 Guided。

- **Quick**（默认）: 最少提问（≤ 3 个），直接产出，标注 `[紧急模式-假设清单]`
- **Expert**: 跳过确认，产出附风险清单和"正常模式下应补充的内容"

## 执行流程

### Step 1: 理解上下文（30 秒）

从用户输入中提取：
- 讨论的主题是什么？
- 要澄清什么？
- 时间紧迫程度？

### Step 2: 生成问题（2 分钟）

按 4 个维度组织问题：

**维度 1: 范围与目标（2 题）**
1. [关于范围的核心问题]
   - 为什么问这个：[帮助界定问题边界]

**维度 2: 用户与场景（2 题）**
3. [关于目标用户的问题]
   - 为什么问这个：[确保方案针对正确人群]

**维度 3: 约束与条件（2 题）**
5. [关于约束的问题]
   - 为什么问这个：[在现实条件内设计解决方案]

**维度 4: 成功标准（1 题）**
7. [如何定义成功]
   - 为什么问这个：[对齐期望和验收标准]

### Step 3: 提供快速回答模板（30 秒）

```
1. [问题 1 简述]: [回答]
2. [问题 2 简述]: [回答]
3. [问题 3 简述]: [回答]
...
```

### Step 4: 推荐下一步

根据回答方向，推荐合适的 PM Skill：
- 需求不清晰 → **pm-problem-frame**
- 需要写文档 → **pm-prd**
- 需要排优先级 → **pm-rice**
- 需要做决策 → **pm-decision**
## 交付前检查

## Hard Bans（硬禁令）

以下模式在所有保真度级别均禁止出现。检测到时立即修正。

| # | 禁止模式 | 修正动作 |
| --- | --------- | -------- |
| HB-1 | 评审/回顾只说好话无改进建议 | 补充 >=3 个改进建议并标注 P0/P1/P2 |
| HB-2 | 行动项无负责人或无截止日期 | 每个行动项必须填入负责人和截止日期 |
| HB-3 | 报告/输出超过 1 页且无摘要 | 压缩至 1 页关键信息，详细内容移入附录 |
| HB-4 | 评分/排序中某维度全给满分（无区分度） | 至少有一个维度的得分范围跨越 >=2 个分值 |
| HB-5 | 跳过问题定义直接给方案 | 先输出 1 句话问题概括，经用户确认后再给方案 |


- [ ] 问题 ≤ 7 个
- [ ] 每个问题有"为什么问这个"
- [ ] 提供了快速回答模板
- [ ] 问题具体不模糊
- [ ] 推荐了下一步行动

## 输出规范

简洁 Markdown，直接输出（不包裹代码块）：
1. **上下文**（1 句话概括理解）
2. **澄清问题**（分 4 组，每组含问题和理由）
3. **快速回答模板**（复制粘贴即填）
4. **推荐下一步**

## 后续推荐

回答完成后根据方向路由到对应 Skill。本 Skill 设计为"入口级"，用完即走。
