# 解决方案简报执行指南

## 步骤详解与示例

### Step 1: 回顾问题

**目标**: 用 2-3 句话总结问题核心

**包含要素**:
- 谁在经历问题
- 问题是什么
- 为什么现在重要

**示例**:
```
新用户在引导过程中流失率达65%，主要因为他们无法快速理解如何使用核心功能。
这是我们最大的获客漏斗，估计每月浪费5万元获客成本。
```

**常见错误**:
- ❌ 重新解释完整问题陈述（太长）
- ❌ 只说"有问题"但不说明具体影响
- ❌ 忘略时机和紧迫性

---

### Step 2: 描述提议方案

**目标**: 用 2-3 段非技术语言解释方案

**包含**:
- 高层次方案
- 如何解决问题
- 关键用户体验变化

**避免**:
- 技术实施细节
- 数据库 schema
- API 规格
- 非技术术语

**示例**:
```
我们将引入一个交互式引导，在首次登录时激活。引导会突出显示驱动激活的三个核心功能，
使用提示和交互式演练。用户可以跳过引导，但会看到持久的"帮助"按钮重新激活。
```

**更多示例**: 见下文"完整示例"章节

---

### Step 3: 列出关键功能

**目标**: 识别 3-5 个核心功能（最小集合）

**原则**: 抵制包含 nice-to-haves 的冲动

**功能描述模板**:
```
| 功能名称 | 简要描述 |
| --- | --- |
| 交互式演练 | 核心功能的分步指南 |
| 智能触发 | 基于用户行为模式激活引导 |
| 渐进披露 | 随用户推进展示功能 |
| 持久帮助 | 随时可访问的帮助按钮和提示 |
```

**检查清单**:
- [ ] 这些功能是否真正解决问题最小集合？
- [ ] 移除任何一个功能是否影响核心价值？
- [ ] 是否有 nice-to-haves 混入？

---

### Step 4: 定义成功指标

**目标**: 3-5 个具体、可衡量、有时限的指标

**指标格式**:
```
| 指标 | 基线 | 目标 | 时间框架 |
| --- | --- | --- | --- |
| 激活率 | 35% | 50% | 30天 |
| 首价值时间 | 4天 | 2天 | 30天 |
```

**指标类型选择**:
- **业务指标**: 收入、转化、留存
- **用户指标**: 激活、参与度、满意度
- **技术指标**: 性能、可靠性
- **护栏指标**: 确保不造成其他损害

---

### Step 5: 承认权衡

**目标**: 明确说明不做什么及原因

**权衡格式**:
```
| 不做什么 | 为什么 |
| --- | --- |
| 不做个性化 | 初始版本通用；v2 基于细分数据个性化 |
| 不支持移动 | MVP 仅 Web；Q2 在 Web 验证后支持 |
```

**权衡原则**:
- 诚实说明当前版本的限制
- 解释为什么这些是正确的权衡
- 如适用，说明何时会加入

---

### Step 6: 识别风险和缓解

**目标**: 揭示顶级风险及应对计划

**风险格式**:
```
| 风险 | 可能性 | 影响 | 缓解措施 |
| --- | --- | --- | --- |
| 用户觉得引导烦人 | 高 | 高 | 使引导可跳过且易于关闭 |
| 工作量高于估算 | 中 | 中 | 使用现有引导 SDK；分阶段推出 |
| 未能改善激活 | 中 | 高 | 全面推出前 A/B 测试对照 |
```

**风险评估提示**:
- 可能性: 高/中/低
- 影响: 高/中/低
- 优先处理高影响 x 高可能性的风险

---

### Step 7: 概述下一步

**目标**: 3-5 个具体、可执行的行动

**下一步格式**:
```
| 行动 | 负责人 | 日期 |
| --- | --- | --- |
| 确定成功指标和发布标准 | 产品 | [日期] |
| 创建低保真模型 | 设计 | [日期] |
| 评估引导 SDK 选项 | 工程 | [日期] |
| 审查并批准解决方案简报 | 利益相关者 | [日期] |
| 创建详细 PRD | 产品 | [日期] |
```

**行动原则**:
- 每个行动有明确负责人
- 每个行动有具体日期
- 行动按优先级排序

## 完整示例

### 示例: 新用户引导解决方案简报

#### 问题回顾

新用户激活率从 35% 下降至 28%，主要原因是首次引导流程过于复杂。这是我们最大的获客漏斗，估计每月浪费 5 万元获客成本。

#### 提议方案

我们将引入一个交互式引导，在首次登录时激活。引导会突出显示驱动激活的三个核心功能（创建、分享、分析），使用提示和交互式演练。用户可以随时跳过引导，但会看到持久的"帮助"按钮重新激活。

#### 关键功能

1. **交互式演练**: 核心功能的分步指南
2. **智能触发**: 基于用户行为模式激活引导
3. **渐进披露**: 随用户推进展示功能
4. **持久帮助**: 随时可访问的帮助按钮

#### 成功指标

| 指标 | 基线 | 目标 | 时间框架 |
|------|------|------|----------|
| 激活率 | 28% | 40% | 上线后 30 天 |
| 首价值时间 | 4 天 | 2 天 | 上线后 30 天 |
| 引导完成率 | - | 60% | 上线后 30 天 |
| 功能采用 | - | +40% | 第 1 周 |

#### 权衡

| 不做什么 | 为什么 |
|---|---|
| 不做个性化 | 初始版本通用；v2 基于细分数据个性化 |
| 不支持移动 | MVP 仅 Web；Q2 在 Web 验证后支持 |
| 不覆盖高级功能 | 引导仅关注核心；为保持简洁排除 |

#### 风险和缓解

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| 用户觉得引导烦人 | 高 | 高 | 使引导可跳过且易于关闭 |
| 工作量高于估算 | 中 | 中 | 使用现有引导 SDK；分阶段推出 |
| 未能改善激活 | 中 | 高 | 全面推出前 A/B 测试对照 |

#### 下一步

| 行动 | 负责人 | 日期 |
|------|--------|------|
| 确定成功指标和发布标准 | 产品 | 1 周 |
| 创建低保真模型 | 设计 | 2 周 |
| 评估引导 SDK 选项 | 工程 | 1 周 |
| 审查并批准解决方案简报 | 利益相关者 | 2 周 |
| 创建详细 PRD | 产品 | 3 周 |
