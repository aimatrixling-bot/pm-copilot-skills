# 文档管道执行指南

> 批量文档生成的执行规则和一致性保障。

---

## 1. 一致性检查规则

### 每个文档产出后必查

| 检查项 | 规则 | 不一致时 |
|--------|------|----------|
| 功能范围 | 路线图 ⊇ PRD ⊇ 技术规格 | 以 PRD 为基准修正 |
| 目标对齐 | 路线图目标 = PRD Why Now | 以 PRD 为基准 |
| 指标对齐 | PRD 成功指标 ⊆ 发布监控指标 | 补充发布计划的监控指标 |
| 时间对齐 | 路线图里程碑 ⊇ 发布计划时间 | 以路线图为基准 |
| 优先级对齐 | RICE 排序 = PRD Feature Scope 优先级 | 以 RICE 为基准 |

### 不一致处理优先级

```
PRD > 路线图 > 技术规格 > 发布计划
```

PRD 是基准文档。其他文档与 PRD 矛盾时，修改其他文档。

---

## 2. 文档间字段映射

| 源文档 | 字段 | 目标文档 | 字段 |
|--------|------|----------|------|
| 路线图 | 里程碑目标 | PRD | Why Now |
| 路线图 | 功能列表 | PRD | Feature Scope |
| PRD | Feature Scope | 技术规格 | 系统范围 |
| PRD | 验收标准 | 测试计划 | 测试用例来源 |
| PRD | 成功指标 | 发布计划 | 监控指标 |
| 技术规格 | API 设计 | 工程需求 | 技术约束 |
| 技术规格 | 数据模型 | 工程需求 | 依赖关系 |

---

## 3. 需求收集模板

### Phase 1 信息收集清单

| 信息 | 来源 | 必需 |
|------|------|------|
| 产品名称 | 用户输入 | 是 |
| 功能列表 | 用户输入/已有文档 | 是 |
| 目标用户 | 用户输入/已有文档 | 是 |
| 约束条件 | 用户输入 | 是 |
| 时间范围 | 用户输入 | 否（默认 Q） |
| 已有研究 | 文件系统搜索 | 否 |

### 信息不足时的提问（≤ 3 个）

1. "这个产品/功能主要解决谁的什么问题？"
2. "你希望什么时间范围内交付？"
3. "有什么技术或业务约束需要注意？"

---

## 4. 管道执行模式

### Quick 模式（1-2 小时）

- 路线图：Now-Next-Later 格式
- PRD：1 页（Why + Scope + AC + Metrics）
- 技术规格：决策记录（非完整规格）
- 发布计划：检查清单

### Full 模式（半天-1 天）

- 路线图：Timeline + Now-Next-Later 双视图
- PRD：完整 PRD（Problem → Solution → Metrics → Risks）
- 技术规格：完整规格（架构 + API + 数据模型 + 风险）
- 发布计划：完整计划（策略 + 检查清单 + 回滚 + 监控）

---

## 5. 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 不做一致性检查 | 文档互相矛盾 | 每产出 1 个文档检查 1 次 |
| 先写完再检查 | 改 5 个文档 | 写一个查一个 |
| 跳过需求收集 | 文档基于假设 | 先收集 ≥ 4 项必需信息 |
| 技术规格脱离 PRD | 开发做了不需要的 | 技术规格范围 = PRD Scope |
| 发布计划无回滚 | 出问题无法回退 | 回滚方案是必选项 |
