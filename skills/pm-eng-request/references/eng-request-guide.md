# 工程需求编写指南

> 让开发和 PM 对齐的高质量需求单编写方法。

---

## 1. Given-When-Then 编写技巧

### 基本格式

```
Given [前置条件/系统状态]
When [用户操作/触发事件]
Then [预期结果/系统行为]
```

### 示例对比

| 差的 AC | 好的 AC |
|---------|---------|
| "搜索功能正常" | Given 用户在搜索页 When 输入关键词点击搜索 Then 展示匹配结果列表，按相关度排序 |
| "支持导出" | Given 用户在数据页 When 点击"导出 CSV" Then 下载包含全部当前筛选结果的 CSV 文件 |
| "处理错误" | Given 支付接口超时 When 用户点击支付 Then 展示"支付处理中，请稍后查看订单状态"且订单状态为"处理中" |

### 异常场景必须覆盖

| 场景类型 | 示例 |
|----------|------|
| 空状态 | Given 无搜索结果 When 用户搜索 Then 展示"未找到相关内容"提示 |
| 权限不足 | Given 普通用户 When 访问管理页 Then 展示"无权限"提示 |
| 网络异常 | Given 网络断开 When 用户操作 Then 展示离线提示并缓存操作 |
| 边界值 | Given 输入超长文本 When 提交 Then 展示字数限制提示 |

## 2. 复杂度评估参考

### T 恤尺码粗估

| 尺码 | 工作量 | 典型范围 |
|------|--------|----------|
| XS | ≤ 0.5 天 | 配置变更、文案修改 |
| S | 0.5-2 天 | 简单 UI 调整、小功能 |
| M | 2-5 天 | 标准功能模块 |
| L | 5-10 天 | 复杂功能、跨系统集成 |
| XL | > 10 天 | 需要拆分——任何 XL 任务都应拆为 ≤ L |

### 复杂度因素

| 因素 | 增加复杂度 |
|------|------------|
| 涉及新 API 集成 | +1 级 |
| 需要数据库 Schema 变更 | +1 级 |
| 涉及第三方服务 | +1 级 |
| 需要迁移旧数据 | +1 级 |
| 安全/合规要求 | +1 级 |
| 性能要求严格 | +1 级 |

## 3. 跨团队协作

### 依赖类型

| 类型 | 说明 | 处理方式 |
|------|------|----------|
| 技术依赖 | 需要其他团队提供 API/服务 | 提前沟通接口契约 |
| 数据依赖 | 需要其他团队的数据 | 确认数据格式和获取方式 |
| 设计依赖 | 需要设计团队提供设计稿 | 提前排期设计需求 |
| 测试依赖 | 需要 QA 团队测试 | 提前沟通测试范围 |
| 发布依赖 | 需要与其他功能同步发布 | 对齐发布时间窗口 |

### 沟通状态标注

| 状态 | 含义 | 行动 |
|------|------|------|
| ✅ 已确认 | 对方已确认可提供 | 正常推进 |
| ⏳ 待确认 | 已沟通但未获确认 | 跟进催确认 |
| ❌ 阻塞 | 对方无法提供或时间冲突 | 升级或调整方案 |
| 🔜 计划中 | 对方已排期但未开始 | 定期跟进 |

## 4. 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| AC 太笼统 | 开发理解偏差 | 用 Given-When-Then |
| 无异常场景 | 上线后出 bug | 覆盖空状态/权限/网络异常 |
| 不标注假设 | 推断当事实 | 所有推断标 [假设] |
| 复杂度低估 | 排期不准 | 用 T 恤尺码 + 复杂度因素 |
| 依赖未沟通 | 被阻塞 | 提前沟通并标注状态 |
