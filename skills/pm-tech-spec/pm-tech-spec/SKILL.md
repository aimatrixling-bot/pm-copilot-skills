# 技术规格

<SUBAGENT-STOP>
如果是子代理派发执行此 Skill，直接按下方指令执行，不重新加载入口 Skill。
</SUBAGENT-STOP>

从 PRD 提取功能需求，生成技术规格框架。帮助 PM 和开发对齐实现方案。

**输入**: PRD 文件路径或功能描述
**输出**: 技术规格文档

## Iron Law

| 铁律 | 违反后果 |
| --- | --- |
| 无 PRD 关联的技术规格是空中楼阁 | 立即停止——先确认关联的 PRD 后继续 |
| 每个技术决策必须有理由 | 立即停止——补充"为什么选这个"后继续 |
| 非功能需求不可省略 | 立即停止——补充性能/安全/可扩展性要求后继续 |

## 反合理化

| 你可能在想的 | 真相 |
| --- | --- |
| "先写代码，文档后补" | 后补的文档永远和实现不一致 |
| "架构图不需要，太明显了" | 你觉得明显 ≠ 新同事觉得明显 |
| "性能以后再优化" | 以后 = 永远不会。先定义最低要求 |
| "安全交给安全团队" | 安全是设计出来的，不是检查出来的 |
| "这个太简单不需要 spec" | 简单的东西出错时最浪费时间排查 |

## 约束翻译模式

从 PRD 到技术规格到工程需求，约束必须可追踪、不丢失。

```
PRD CONSTRAINTS → IMPLEMENTATION CONTRACT → HANDOFF
    (提取)            (翻译)              (回溯)
```

| 阶段 | 职责 | 产出 |
| --- | --- | --- |
| **CONSTRAINTS**（本 Skill） | 从 PRD 提取所有约束：业务规则、边界条件、合规要求、非功能需求 | 约束清单（每条有 ID） |
| **IMPLEMENTATION CONTRACT**（本 Skill） | 将每条约束翻译为可验证的技术实现要求 | 技术约束矩阵 |
| **HANDOFF**（pm-eng-request） | 每个工程需求必须回溯到技术约束 ID | 约束覆盖率报告 |

**约束分类**（提取时按此分类）：

| 类别 | 示例 | 优先级 |
| --- | --- | --- |
| 业务规则 | "订单金额 > 5000 需主管审批" | P0 |
| 数据约束 | "手机号必须支持国际格式" | P0 |
| 非功能 | "列表加载 ≤ 2s（P95）" | P0 |
| 合规/安全 | "用户数据 30 天后自动删除" | P0 |
| 兼容性 | "支持 iOS 15+" | P1 |
| 边界条件 | "并发下单时库存不超卖" | P1 |

## 执行流程

```
1. Read PRD 提取功能需求和非功能需求
2. 约束提取 — 从 PRD 提取所有约束，每条分配 C-[NNN] ID
   ├── 扫描 PRD 的每个功能点，识别隐含约束
   ├── 检查验收标准中的量化指标
   └── 标注来源 PRD 章节
3. 系统概述 — 定义要构建什么、范围、关键目标
4. 架构设计 — 组件关系、技术栈、交互方式
5. 数据模型 — 实体、字段、关系、索引
6. API 规范 — 端点、请求/响应、认证、错误码
7. 非功能需求 — 性能、安全、可扩展性、监控
8. 约束翻译矩阵 — 每条 C-[NNN] 对应技术实现方案
   ├── 约束 → 技术方案（如何实现）
   ├── 约束 → 验证方法（如何确认已实现）
   └── 未覆盖约束 → 标注 [待覆盖]
9. 技术决策 — 选项、选择、理由（记录为什么）
10. Iron Law 检查
    ├── 有 PRD 关联？
    ├── 每个技术决策有理由？
    ├── 非功能需求已定义？
    └── 约束翻译覆盖率 = 100%？（无 [待覆盖]）
11. 标注 [待开发确认] 的技术细节
12. 产出技术规格文档
```

## Hard Bans（硬禁令）

以下模式在所有保真度级别均禁止出现。检测到时立即修正。

| # | 禁止模式 | 修正动作 |
| --- | --------- | -------- |
| HB-1 | 风险缓解措施为"持续监控"、"加强沟通"等不可执行描述 | 替换为：谁在什么时间点检查什么指标，达到什么阈值触发什么动作 |
| HB-2 | 架构图/组件关系中存在无对应 API 的调用 | 补充 API 定义，或修正组件关系 |
| HB-3 | 没有标注来源的定量断言 | 补充数据来源，或改为定性描述并标注 [待验证] |
| HB-4 | 数据模型缺少字段约束（类型/必填/默认值） | 补充每个字段的完整约束 |
| HB-5 | 集成方案含"待定"/"TBD" | 强制明确集成方式，不确定的列入 Future 并标注决策条件 |

## 交付前检查

- [ ] 关联 PRD 已标注
- [ ] 覆盖 PRD 中的所有 P0 功能
- [ ] API 设计包含端点、请求/响应格式、错误码
- [ ] 数据模型含实体、关系、索引
- [ ] 非功能需求已考虑（性能、安全、可用性）
- [ ] 技术选型有理由说明
- [ ] 风险和缓解已列出
- [ ] [待开发确认] 项已标注
- [ ] 约束翻译矩阵覆盖所有 P0 约束（无 [待覆盖]）
- [ ] 每条约束有验证方法（可测试）

## 常见错误

| 错误 | 后果 | 正确做法 |
| --- | --- | --- |
| 过度设计 | 实施延迟 | 从简单设计开始，随需求演进 |
| 缺少非功能需求 | 只关注功能忽略性能/安全 | 始终定义性能、安全要求 |
| API 不规范 | 前后端对接困难 | 使用统一 API 规范 |
| 无迁移计划 | 现有系统切换出问题 | 包含迁移和回滚计划 |
| 无错误处理 | 异常场景未定义 | 定义所有错误码和消息 |
