# 技术规格质量检查清单

> 确保每个技术规格都符合 PM OS 的 Spec-Driven 标准

---

## 核心检查项 (必须全部通过)

### 概述和范围

- [ ] **目的清晰**: 系统目的明确，说明做什么和为什么
- [ ] **范围明确**: In Scope 和 Out Scope 清晰定义
- [ ] **关键目标**: 性能、可扩展性、可靠性目标具体可测量

### 架构设计

- [ ] **系统图完整**: Mermaid 图清晰展示组件和交互
- [ ] **组件定义**: 每个组件有明确的职责和技术栈
- [ ] **接口定义**: 组件间接口清晰定义
- [ ] **交互流程**: 关键流程有序列图或流程图
- [ ] **架构合理**: 架构不过度设计，符合当前需求

### 数据模型

- [ ] **实体完整**: 所有实体字段定义完整
- [ ] **类型正确**: 字段类型选择合理 (bigint 主键, datetime 时间戳)
- [ ] **索引策略**: 为常用查询字段建立索引
- [ ] **关系清晰**: 实体间关系 (1:1, 1:N, M:N) 明确
- [ ] **ERD 图**: 包含关系图或实体关系说明
- [ ] **软删除**: 使用 deleted_at 而非物理删除

### API 规范

- [ ] **端点完整**: 所有端点列入清单
- [ ] **请求格式**: 每个端点有请求参数说明
- [ ] **响应格式**: 每个端点有响应示例
- [ ] **错误处理**: 定义所有错误代码和消息
- [ ] **认证**: 说明认证方式
- [ ] **版本控制**: 包含 API 版本策略 (如适用)

### 非功能需求

- [ ] **性能指标**: 响应时间、吞吐量、并发有具体目标值
- [ ] **可扩展性**: 说明水平扩展能力
- [ ] **缓存策略**: 定义缓存方案
- [ ] **安全**: 认证、授权、加密、合规要求完整
- [ ] **可靠性**: 可用性目标、部署策略、回滚计划
- [ ] **监控**: 指标、告警、日志有定义

### 技术决策

- [ ] **决策记录**: 关键技术决策有记录
- [ ] **选项分析**: 考虑了多个选项
- [ ] **理由充分**: 选择有充分的理由说明
- [ ] **后果评估**: 评估决策的影响

### 实施考虑

- [ ] **阶段清晰**: 开发有明确的阶段划分
- [ ] **依赖识别**: 技术和业务依赖已识别
- [ ] **迁移计划**: 如适用，有迁移和回滚计划
- [ ] **技术债务**: 承认技术债务并有偿还计划

### 测试策略

- [ ] **测试类型**: 单元、集成、E2E、性能、安全测试计划
- [ ] **覆盖率**: 有覆盖率目标
- [ ] **测试数据**: 说明测试数据策略

### 文档质量

- [ ] **遵循规范**: 遵循 TEMPLATE-ENGINE.md 五段式结构
- [ ] **TL;DR 有效**: 3句话概括核心
- [ ] **图表使用**: 复杂架构有 Mermaid 图
- [ ] **表格使用**: 数据使用表格呈现
- [ ] **残酷风险区**: 包含 Trade-offs 分析
- [ ] **下一步明确**: DPER Loop 包含具体行动项

## 按复杂度分级

### 简单功能规格

**适用**: 单一功能，无复杂架构

必须包含:
- [ ] 概述和范围
- [ ] 数据模型
- [ ] API 规范
- [ ] 残酷风险区
- [ ] 下一步行动

可选包含:
- [ ] 简化的架构图
- [ ] 基本性能要求

### 中等复杂度规格

**适用**: 多组件交互，有一定复杂度

必须包含简单功能全部，加上:
- [ ] 完整架构设计
- [ ] 交互流程图
- [ ] 非功能需求
- [ ] 技术决策记录

### 高复杂度规格

**适用**: 新系统、重构、关键基础设施

必须包含中等复杂度全部，加上:
- [ ] 详细的分阶段实施计划
- [ ] 迁移和回滚策略
- [ ] 完整测试策略
- [ ] 依赖关系矩阵
- [ ] 监控和告警详细方案

## 常见问题检查

### 架构设计

❌ **过度设计**
```
为简单 CRUD 设计微服务架构、事件溯源、CQRS...
```
→ YAGNI 原则，从简单开始

✅ **适度设计**
```
单体应用 + 清晰分层，必要时可拆分
```

❌ **缺少图表**
```
纯文字描述架构
```
→ 难以理解

✅ **可视化架构**
```
使用 Mermaid 图展示组件和交互
```

### 数据模型

❌ **错误主键类型**
```
id: int (可能溢出)
```

✅ **正确主键类型**
```
id: bigint (防止溢出)
```

❌ **缺少时间戳**
```
用户表只有 id, name, email
```

✅ **标准字段**
```
id, name, email, created_at, updated_at, deleted_at
```

❌ **物理删除**
```
DELETE FROM users WHERE id = ?
```

✅ **软删除**
```
UPDATE users SET deleted_at = NOW() WHERE id = ?
```

### API 设计

❌ **不一致的错误格式**
```
{"error": "Not found"}
{"message": "User does not exist"}
```

✅ **统一错误格式**
```
{"success": false, "error": {"code": "NOT_FOUND", "message": "..."}}
```

❌ **缺少版本控制**
```
POST /api/users
```

✅ **包含版本**
```
POST /api/v1/users
```

### 非功能需求

❌ **模糊性能要求**
```
"系统应该快"
```

✅ **具体性能指标**
```
P50 响应时间 < 100ms
P99 响应时间 < 500ms
```

❌ **缺少安全考虑**
```
无认证、授权说明
```

✅ **完整安全方案**
```
认证: JWT Bearer Token
授权: RBAC 权限控制
传输: TLS 1.3
存储: 敏感字段 AES-256 加密
```

## PM OS 特殊要求

### Spec-Driven 原则

- [ ] **可执行性**: 规格足够详细，可直接指导开发
- [ ] **可验证性**: 每个需求可测试验证
- [ ] **可追溯性**: 技术决策可追溯到业务需求

### Brutally Honest 原则

- [ ] **技术债务暴露**: 不隐藏已知技术问题
- [ ] **假设明确**: 说明技术假设及其风险
- [ ] **权衡说明**: 残酷风险区说明技术权衡

### Professional & Prudent 原则

- [ ] **适度设计**: YAGNI，避免过度设计
- [ ] **演进思维**: 架构支持未来演进但不过度设计
- [ ] **务实选择**: 技术选择基于实际情况

## 评审流程

### 自查 (技术负责人)

- [ ] 完成核心检查项
- [ ] 运行常见问题检查
- [ ] 确认复杂度级别要求

### 架构评审

- [ ] 架构合理性通过
- [ ] 技术选择有理有据
- [ ] 非功能需求现实可达
- [ ] 风险有缓解措施

### 安全评审

- [ ] 认证授权方案完善
- [ ] 数据保护符合合规要求
- [ ] 安全漏洞有考虑

### 性能评审

- [ ] 性能目标可测量
- [ ] 性能测试计划可行
- [ ] 扩展策略合理

## 质量评分

| 等级 | 标准 | 行动 |
|------|------|------|
| **A** | 所有核心检查项通过 | 可进行开发 |
| **B** | 1-2 项次要检查项未通过 | 修补后开发 |
| **C** | 3+ 项检查项未通过 | 重大修订 |
| **D** | 架构设计不合理 | 重新设计 |
