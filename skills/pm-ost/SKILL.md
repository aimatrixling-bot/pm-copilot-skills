# 开放技能树分析

<SUBAGENT-STOP>
如果是子代理派发执行此 Skill，直接按下方指令执行，不重新加载入口 Skill。
</SUBAGENT-STOP>

识别 PM 能力分布，找到成长方向。知道自己不会什么比知道自己会什么更重要。

**输入**: 分析对象（个人/团队）+ 自我评估
**输出**: 技能雷达 + 成长建议

## Iron Law

| 铁律 | 违反后果 |
| --- | --- |
| 自评必须有证据支撑 | 立即降级——每项 ≥ 3 分的能力必须有具体事例 |
| 差距 > 2 分的维度必须有行动计划 | 立即补充——没有计划的识别 = 抱怨 |
| 不超过 3 个优先提升维度 | 删减——什么都想提升 = 什么都提升不了 |

## 反合理化

| 你可能在想的 | 真相 |
| --- | --- |
| "我什么都还行" | "还行" = 没有突出能力。找到最强的 1-2 个 |
| "差距太多，先提升最重要的" | 对的！但只选 ≤ 3 个。超过 3 个 = 没有 focus |
| "技能树太理论了" | 每个维度对应真实的 PM 工作场景。不会 = 做不好 |
| "我现在就挺好的" | 市场在变。停止成长 = 开始落后 |
| "需要别人评估才准确" | 自评是起点。360 评估是验证。先自评 |

## PM 技能树（6 维度）

| 技能 | 描述 | 评估维度 |
| --- | --- | --- |
| 需求分析 | 理解和定义用户需求 | 问题定义、用户研究、数据驱动 |
| 产品设计 | 设计解决方案 | 用户体验、交互设计、信息架构 |
| 项目管理 | 推进项目交付 | 优先级、排期、风险管理 |
| 沟通协作 | 与团队协作 | 文档、演示、跨团队协调 |
| 数据分析 | 基于数据决策 | 指标定义、A/B 测试、数据解读 |
| 战略思维 | 长期规划 | 市场洞察、竞品分析、商业模型 |

## 执行流程

```
1. 了解分析对象（个人/团队）
2. 引导自评（6 维度 1-5 分）
3. 要求证据（≥ 3 分的维度列举具体事例）
4. 生成能力雷达
5. 识别差距最大的 2-3 个维度
6. 匹配 PM Copilot Skills 练习建议
7. 产出成长计划（≤ 3 个提升项）
8. Iron Law 检查
   ├── 自评有证据？
   ├── 差距有计划？
   └── 提升项 ≤ 3 个？
```

## 评分标准

| 分数 | 含义 | 证据要求 |
| --- | --- | --- |
| 1 | 了解概念 | 能描述该技能的定义 |
| 2 | 有基本实践 | 做过 1-2 次，需要指导 |
| 3 | 能独立完成 | 有具体产出物（文档/案例） |
| 4 | 能教别人 | 带过其他人完成该技能 |
| 5 | 有方法论创新 | 总结出自己的方法论 |

## 交付前检查

- [ ] 6 维度均有评分
- [ ] ≥ 3 分的维度有证据
- [ ] 差距最大的 ≤ 3 个维度已识别
- [ ] 每个提升维度有具体行动计划
- [ ] 行动计划关联 PM Copilot Skills

## 常见错误

| 错误 | 后果 | 正确做法 |
| --- | --- | --- |
| 所有维度都打 3-4 分 | 看不出特点 | 诚实评估，允许 1-2 分 |
| 无证据支撑 | 自评不可信 | ≥ 3 分必须有具体事例 |
| 提升项太多 | 无法聚焦 | 最多 3 个优先提升维度 |
| 计划太模糊 | 无法执行 | 具体 Skill + 练习方式 + 时间 |
| 只关注短板 | 忽略长板优势 | 先强化长板，再补短板 |
