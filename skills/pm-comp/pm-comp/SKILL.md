# 竞品分析

<SUBAGENT-STOP>
如果是子代理派发执行此 Skill，直接按下方指令执行，不重新加载入口 Skill。
</SUBAGENT-STOP>

引导 PM 进行系统化的竞品分析——从"我觉得竞品怎么样"到"数据表明竞品的优劣"。

**核心原则**：竞品分析的目的不是模仿，是找到差异化空间。好的竞品分析告诉你"不该做什么"和"可以做得更好"。

## Iron Law（铁律）

| 铁律 | 违反后果 |
| --- | --- |
| 每个结论必须有来源 | 立即停止——补充来源后继续，无法补充的标注 [未验证] |
| 竞品信息必须标注时效 | 立即停止——补充数据收集日期后继续 |
| 不可全盘肯定或否定竞品 | 立即停止——每个竞品必须列出 ≥ 1 个优势和 ≥ 1 个劣势 |

## 反合理化

| 你可能在想的 | 真相 |
| --- | --- |
| "我了解这个竞品，不需要查" | 你了解的是印象，不是数据。印象会过时 |
| "竞品做得很好，我们学它" | 竞品的好 ≠ 适合你的用户。先验证用户需求 |
| "没有竞品数据" | 没有公开数据 ≠ 没有信息。可以分析官网、定价页、用户评价 |
| "小竞品不用看" | 今天的忽略 = 明天的威胁。小竞品可能代表趋势 |
| "这个数据足够了" | 单一来源的数据不可靠。核心结论至少 2 个来源 |

## Entry Mode

### Guided（引导模式）— 预计 6-8 个问题

**Step 1: 分析目标**
问："这次竞品分析的目的是什么？请选择：
A. 了解市场格局 B. 找到差异化空间 C. 评估新功能可行性 D. 定价参考 E. 其他"
→ 记录分析目标

**Step 2: 竞品范围**
问："你想分析哪些竞品？（建议 3-5 个，最多 8 个）
类型：A. 直接竞品 B. 间接竞品 C. 替代方案 D. 国际对标"
→ 记录竞品列表 + 类型

**Step 3: 分析维度**
问："最关注哪些维度？请选择（可多选）：
A. 功能对比 B. 定价策略 C. 用户体验 D. 技术架构 E. 市场定位 F. 用户口碑 G. 增长策略"
→ 记录分析维度（≥ 3 个）

**Step 4: 功能清单**
问："列出你想对比的核心功能（5-15 个）"
→ 建立功能对比矩阵的行

**Step 5: 数据收集**
问："你有这些竞品的第一手使用体验吗？或者需要我帮你搜索公开信息？"
→ A. 用户提供 → 标注 [用户经验] | B. AI 搜索 → 标注来源 + [搜索日期]

**Step 6: 评分对比**
基于 Step 4-5 的数据，对每个竞品×维度评分 → 展示对比矩阵 → 用户确认/修正

**Step 7: 差异化分析**
问："基于对比，你看到哪些差异化机会？"
→ 引导总结差异点 → 用户确认

**Step 8: 产出确认**
展示完整竞品分析报告 → 用户最终确认 → 交付

### Quick（快速模式）— ≤ 2 个问题后直接产出

**问题 1**: "竞品名称或行业？你的产品是什么？"
**问题 2**（可选）: "你最关心哪个维度？（功能/定价/用户体验）"

然后：
- WebSearch 搜索竞品公开信息
- 自动选择 5 个标准维度对比
- 产出竞品分析表格
- 所有来源标注引用
- 自动生成差异化建议
- AI 搜索数据标注 [搜索日期]，置信度默认"中"

## Scope Gate（范围门控）— ⚠️ 在任何执行之前必须先通过

**此检查在模式判断、竞品搜索、任何输出之前执行。违反以下任一规则时，不得进入任何模式，不得生成任何竞品分析内容。**

### 硬拒绝（立即停止，不生成任何分析内容）

| 场景 | 信号词/特征 | 正确响应 |
| --- | --- | --- |
| 要求制造偏向性分析 | "证明我们更好"、"证明比所有竞品都好"、"找出竞品的弱点"、"写一份给投资人看的我们更厉害的分析" | **硬拒绝**："竞品分析的价值在于客观中立。'证明我们更好'不是分析，是为结论找证据——投资人一眼就能看穿。如果你需要一份让投资人信服的竞争力展示，我建议做诚实的 SWOT 分析，突出你的差异化优势而不是否认竞品的存在。" |
| 分析对象无公开信息 | "秘密公司"、"未公布"、" stealth"、"只有内部知道" | **硬拒绝**："没有公开信息就无法进行有意义的竞品分析——任何分析都是虚构。如果你有具体的竞品名称和公开产品，我可以开始分析。或者如果你正在做一款新产品想了解竞品格局，请告诉我方向。" |
| 要求获取非公开内部数据 | "内部数据"、"机密"、"竞品的营收"、"黑进" | 拒绝 → 只提供公开信息 → 说明数据局限性 |
| 恶意用途 | "攻击竞品"、"竞品服务器"、"搞垮" | 明确拒绝 |

**硬拒绝的执行方式**：
1. **第一句话就拒绝**，不说"我理解你"或"好的"
2. 解释为什么拒绝（1-2句话）
3. 提供 1 个可行的替代方向
4. **不生成任何竞品分析内容、不搜索任何竞品信息**

### 软质疑（先指出问题，用户确认后才继续）

| 场景 | 质疑方向 |
| --- | --- |
| 对比完全不同品类的产品 | 指出不可比性 → 从 PM 工作流角度讨论互补关系 |

### 判断流程

```
收到请求
  → 硬拒绝检查（第1优先级）
    → 匹配硬拒绝 → 立即拒绝 + 解释 + 替代方向 → 结束（不进入任何模式）
    → 不匹配 → 软质疑检查
      → 匹配 → 先指出问题 → 用户确认后才继续
      → 不匹配 → 正常进入模式判断
```

**关键原则**：
- **先拒绝再讨论**，不是"先指出问题然后继续执行"
- 偏向性请求不给"诚实版"选项——直接拒绝
- 拒绝的第一句话必须是否定，不能以肯定开头

## 执行流程

```
触发 pm-comp
    ├── 1. 模式判断（Guided/Quick）
    ├── 2. 确定竞品范围（用户指定或 WebSearch 发现）
    ├── 3. 收集数据
    │     ├── WebSearch 搜索公开信息
    │     ├── WebFetch 深度阅读关键页面
    │     └── Read 项目已有竞品文档
    ├── 4. 构建对比矩阵（竞品 × 维度 评分）
    ├── 5. Iron Law 检查
    │     ├── 每个结论有来源？
    │     ├── 数据有时效标注？
    │     └── 每个竞品有优劣势？
    ├── 6. 差异化分析
    │     ├── 识别空白区域
    │     ├── 识别差异化机会
    │     └── 识别学习机会
    └── 7. 交付 + 建议下一步（→ pm-prd 或 → pm-rice）
```

## Hard Bans（硬禁令）

以下模式在所有保真度级别均禁止出现。检测到时立即修正。

| # | 禁止模式 | 修正动作 |
| --- | --------- | -------- |
| HB-1 | 结论没有数据来源或标注为常识 | 补充来源；无法补充的标注 [未验证] 并降级置信度 |
| HB-2 | 相关性表述暗示因果关系（"A 增长导致 B 下降"） | 改为"与...相关"并标注 [相关性非因果] |
| HB-3 | 竞品只写优势或只写劣势 | 每个竞品必须 >=1 个优势 + >=1 个劣势 |
| HB-4 | 竞品数据超过 12 个月未标注时效 | 标注 [数据过旧-待更新] |
| HB-5 | 单一数据源支撑核心结论 | 核心结论至少 2 个独立来源，或标注 [单一来源-置信度低] |

## 交付前检查

- [ ] 竞品范围明确（3-8 个，含类型分类）
- [ ] 分析维度 ≥ 3 个
- [ ] 对比矩阵完整（竞品 × 维度）
- [ ] 每个结论有来源引用
- [ ] 数据收集日期已标注
- [ ] 每个竞品有 ≥ 1 个优势和 ≥ 1 个劣势
- [ ] 置信度标注合理（不可全"高"）
- [ ] 差异化建议已列出
- [ ] **差异化建议中的技术路径已验证**（非 Magic Step）
- [ ] **引用的竞品功能有数据源支撑**（不是假设存在）
- [ ] 所有定量断言已标注来源（无来源的标注 [待验证]）
- [ ] 推断和建议已标注所依据的事实
- [ ] 核心结论有 >=2 个独立来源，或已标注 [单一来源-置信度低]

## Source Attribution（来源标注规范）

### 断言分类

| 类型 | 标注方式 | 示例 |
|------|---------|------|
| **事实** | `[来源: URL/文档名]` | "Notion 拥有 3000 万用户 [来源: TechCrunch 2024]" |
| **推断** | `[推断: 依据→结论]` | "竞品 X 可能拓展此市场 [推断: 近期招聘该领域人才→扩张意图]" |
| **建议** | `[建议: 基于...]` | "建议优先考虑差异化策略 [建议: 基于 3 个竞品均未覆盖用户群 Y]" |
| **未验证** | `[待验证]` | "用户留存率约 60% [待验证]" |

### 检查规则

- 所有定量数据（数字、百分比、金额）必须有来源标注
- 来源时效超过 12 个月标注 `[数据过旧-待更新]`
- 核心结论至少 2 个独立来源支撑
- 推断和建议必须标注所依据的事实

## 输出结构

**格式铁律**：竞品分析报告作为文档类产出，在对话中直接输出为 Markdown。不得包含 `<generative-ui-widget>`、`<style>` 等无法在 Markdown 渲染中正确显示的标签。不将报告包裹在代码围栏（```` ```markdown ``` ````）中。所有可视化内容（表格、列表、流程）用 Markdown 原生语法表达。若用户要求导出为 docx/pdf 等格式，按目标格式处理。

完整的竞品分析报告使用 `references/output-template.md` 中的模板。核心产出：

1. **竞品概览表** — 竞品/类型/定位/核心优势
2. **详细分析**（每个竞品）— 功能/定价/UX/SWOT
3. **对比矩阵** — 功能对比表 + 定位对比表
4. **差异化机会** — 空白区域 + 服务不足的细分
5. **战略建议** — 要构建什么 + 要避免什么
6. **残酷风险区** — 分析局限 + 过时风险

质量检查标准详见 `references/quality-checklist.md`。
