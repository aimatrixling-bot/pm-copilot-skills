## 1. Build-Measure-Learn 循环

**来源**: Ch 1-2, p61-133 | **PM 相关度**: ⭐⭐⭐

```
Build-Measure-Learn 反馈循环

         ┌──────────────────────────────┐
         │                              │
         │    ┌───────────┐             │
         │    │   Learn   │ ← 验证假设  │
         │    │  （学习）  │   获取认知  │
         │    └─────┬─────┘             │
         │          │                   │
         │    ┌─────▼─────┐             │
         │    │  Measure  │ ← 量化结果  │
         │    │ （度量）  │   创新核算  │
         │    └─────┬─────┘             │
         │          │                   │
         │    ┌─────▼─────┐             │
         │    │   Build   │ ← 构建 MVP  │
         │    │ （构建）  │   最小成本  │
         │    └───────────┘             │
         │                              │
         └──────────────────────────────┘

注意：实际执行顺序是逆向的
    想法 → 确定 Learn 目标 → 设计 Measure 方式 → 构建 Build 对象
```

**核心原则**：循环速度是关键——越快完成一次完整循环，学习效率越高。

## 2. 最小可行产品（MVP）

**来源**: Ch 3, 6, p135-172, 320-400 | **PM 相关度**: ⭐⭐⭐

```
MVP 设计原则

核心定义：用最小成本验证核心假设的产品版本

MVP 的形态（从简到繁）：
├── 视频 MVP：用演示视频验证需求（Dropbox 案例）
├── 烟雾测试：Landing Page + 注册按钮，无实际产品
├── 礼宾 MVP：人工模拟产品功能（Food on the Table）
├── 绿野仙踪测试：前面是人，后面是人工（Wizard of Oz）
├── 单功能 MVP：只做一个功能验证（Groupon 案例）
└── 预售 MVP：先卖后做（Zappos 案例）

MVP ≠ 半成品：
├── MVP 有明确的学习目标
├── MVP 可以为早期客户创造价值
├── MVP 不是"偷工减料"，而是"精准投入"
└── MVP 之后必须有明确的度量计划
```

**经典案例**：Zappos 用手机拍鞋店照片验证"人们是否愿意在线买鞋"；Dropbox 用3分钟演示视频获得75000个beta注册。

## 3. 经证实的认知（Validated Learning）

**来源**: Ch 3, p135-203 | **PM 相关度**: ⭐⭐⭐

```
经证实的认知 vs 其他"学习"

┌────────────────────────────────────────────────┐
│  经证实的认知 (Validated Learning)              │
│  ├── 以实验数据为基础                            │
│  ├── 可以被投资人/团队客观评估                    │
│  ├── 用于指导下一轮 Build-Measure-Learn 循环      │
│  └── 是创业进步的核心度量单位                      │
├────────────────────────────────────────────────┤
│  不是经证实认知的"学习"：                        │
│  ├── "我们学到了很多"（无具体数据支撑）            │
│  ├── "市场还没准备好"（失败的借口）                │
│  ├── "我们只是需要更多时间"（拖延转向的借口）      │
│  └── "代码写完了但没上线"（WIP，不是学习）         │
└────────────────────────────────────────────────┘
```

## 4. 创新核算（Innovation Accounting）

**来源**: Ch 7, p402-523 | **PM 相关度**: ⭐⭐⭐

```
创新核算三步骤

Step 1: 建立基准线（Establish Baseline）
├── 用 MVP 测试转化率（注册→激活）
├── 测试客户留存率
├── 测试客户付费率
└── 得到当前真实数据（即使很差）

Step 2: 调优引擎（Tune the Engine）
├── 基于基准线进行 A/B 测试
├── 每次实验都应向目标指标移动
├── 如果指标在改善 → 继续调优
└── 如果指标停滞 → 考虑转向

Step 3: 转向或坚持（Pivot or Persevere）
├── 调优是否产生实质性进步？
├── 是否验证了信仰飞跃假设？
├── 如果否 → 转向
└── 如果是 → 坚持，继续深入

虚荣指标 vs 可行动指标：
┌─────────────────┬──────────────────┐
│ 虚荣指标         │ 可行动指标         │
├─────────────────┼──────────────────┤
│ 总注册用户数     │ 活跃用户占比       │
│ 页面浏览量       │ 转化率            │
│ 下载量          │ 留存率（同期群）    │
│ 客户满意度       │ 付费转化率         │
│ 总收入          │ LTV / CAC 比率     │
└─────────────────┴──────────────────┘

指标三 A 原则：
├── Actionable（可行动）：能指导决策
├── Accessible（可访问）：全员可理解
└── Auditable（可审计）：可追溯数据来源
```

## 5. 转向（Pivot）

**来源**: Ch 8, p525-640 | **PM 相关度**: ⭐⭐⭐

```
10 种转向类型

| 类型 | 说明 | 案例 |
|------|------|------|
| 放大转向 (Zoom-in) | 单一功能变整个产品 | Grockit 的懒注册 |
| 缩小转向 (Zoom-out) | 整个产品变一个功能 | — |
| 客户细分转向 | 目标客户群体改变 | Votizen B2C→B2B |
| 客户需求转向 | 发现真正要解决的问题 | Wealthfront 游戏→金融 |
| 平台转向 | 应用↔平台切换 | — |
| 商业架构转向 | 高毛利↔低毛利切换 | — |
| 价值获取转向 | 变现模式改变 | — |
| 增长引擎转向 | 粘性↔病毒↔付费切换 | IMVU 病毒→付费 |
| 渠道转向 | 分销渠道改变 | — |
| 技术转向 | 技术方案改变 | — |

转向信号：
├── 实验效果持续递减
├── 产品开发"应该更高效"但不知道哪里不对
├── 增长引擎调优呈现边际收益递减
└── 团队士气下降，目标模糊

跑道新定义：
├── 不是"还能活几个月"
└── 而是"还能转向几次"
    延长跑道 = 加快每次转向速度
```

## 6. 信仰飞跃假设（Leap-of-Faith Assumptions）

**来源**: Ch 5, p275-319 | **PM 相关度**: ⭐⭐⭐

```
两大信仰飞跃假设

1. 价值假设（Value Hypothesis）
   └── 客户在使用产品时是否真的获得了价值？
       验证方式：早期用户行为数据（激活率、留存率）
       案例：Facebook 的"10天内加7个好友"

2. 增长假设（Growth Hypothesis）
   └── 新客户将如何发现产品？
       验证方式：获客渠道测试、病毒系数测量
       案例：Hotmail 的邮件签名推广

假设验证工具：
├── 类比（Analogy）：参考相似产品/行业
├── 反类比（Antithesis）：与失败案例对比
├── 客户原型（Customer Archetype）：深入理解目标用户
└── Mark Cook 四问框架：
    1. 客户认识到他们有你需要解决的问题吗？
    2. 如果有，他们会买吗？
    3. 他们会向我们买吗？
    4. 我们能用可持续的方式交付吗？
```

## 7. 三种增长引擎

**来源**: Ch 10, p719-779 | **PM 相关度**: ⭐⭐⭐

```
增长引擎对比

┌──────────┬──────────────┬──────────────┬──────────────┐
│          │  粘性引擎     │  病毒式引擎   │  付费引擎     │
├──────────┼──────────────┼──────────────┼──────────────┤
│ 核心     │ 留存 > 流失   │ 传播自带增长   │ LTV > CAC    │
│ 关键指标 │ 流失率、      │ 病毒系数      │ LTV、CPA、   │
│          │ 复合增长率    │ （临界值 1.0） │ 边际利润     │
│ 增长策略 │ 提升参与度    │ 优化传播机制   │ 提高变现能力  │
│ 典型案例 │ SaaS、订阅    │ Hotmail、     │ 电商、        │
│          │              │ Facebook      │ 企业软件      │
│ 陷阱     │ 漏水桶效应    │ 病毒系数 < 1  │ CPA 通胀      │
└──────────┴──────────────┴──────────────┴──────────────┘

可持续增长四大来源：
├── 口碑传播（满意客户推荐）
├── 产品使用的副作用（使用即曝光）
├── 资助型广告（用收入买增长）
└── 重复购买/使用（订阅/复购）
```

## 8. 小批量与持续部署

**来源**: Ch 9, p642-717 | **PM 相关度**: ⭐⭐

```
小批量 vs 大批量

大批量死亡螺旋：
├── 开发周期越长 → 恐惧感越大
├── 恐惧感越大 → 塞入更多功能"保险"
├── 更多功能 → 周期更长
└── 循环往复，无法交付

小批量优势：
├── 更早发现问题（几乎零返工）
├── 更快的反馈循环
├── 更少的 WIP（在制品）
└── 更高的系统效率

关键概念：
├── 单件流（Single-Piece Flow）：批量=1
├── SMED：快速换模技术
├── 安灯绳（Andon Cord）：发现问题立即停线
├── WIP（在制品）：未发布的设计/假设
└── 持续部署：每次变更自动测试→部署→监控
    IMVU 每天50次发布，"产品免疫系统"自动回滚
```

## 9. Five Whys 根本原因分析

**来源**: Ch 11, p781-885 | **PM 相关度**: ⭐⭐

```
Five Whys 方法

问题 → 为什么？→ 为什么？→ 为什么？→ 为什么？→ 为什么？→ 根本原因

示例：
├── 机器停了 → 为什么？→ 保险丝烧了
├── 保险丝烧了 → 为什么？→ 超载了
├── 超载了 → 为什么？→ 轴承不够润滑
├── 不够润滑 → 为什么？→ 润滑泵坏了
└── 润滑泵坏了 → 为什么？→ 没有定期保养制度

按比例投资策略：
├── 第一次出现：忽略（偶然事件）
├── 第二次出现：轻度投资（5%时间）
├── 第三次出现：中度投资（10%时间）
└── 持续出现：重度投资（25%时间）

自适应组织原则：
├── 每次改进都是渐进的（不搞一次性大改）
├── 由一线团队主导（而非管理层强加）
└── "速度与质量不可交易"——不是二选一
```

## 10. 精益创业五大原则

**来源**: Ch 1-2, p21-133 | **PM 相关度**: ⭐⭐⭐

```
精益创业五大核心原则

1. 创业者无处不在
   └── 创业管理不限于车库创业公司，大企业内部创新同样适用

2. 创业即管理
   └── 创业需要一种专为极端不确定性设计的新型管理学科

3. 经证实的认知
   └── 学习是创业进步的核心度量单位，必须用数据验证

4. 构建-度量-学习
   └── 将所有创业活动转化为科学实验，快速循环

5. 创新核算
   └── 用创新核算替代传统财务核算，量化学习进展
```
