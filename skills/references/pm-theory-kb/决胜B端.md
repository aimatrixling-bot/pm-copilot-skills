## 1. B 端产品建设全流程

**来源**: Ch 3, p81-95 | **PM 相关度**: ⭐⭐⭐

```
B端产品建设六阶段

业务调研 → 整体方案设计 → 细节方案设计 → 技术方案设计 → 项目管理与实施 → 运营迭代
   │            │               │               │               │              │
   │            │               │               │               │              └── 需求管理
   │            │               │               │               │                  迭代优化
   │            │               │               │               │                  数据分析
   │            │               │               │               └── PMO
   │            │               │               │                   敏捷开发
   │            │               │               │                   跨端协调
   │            │               │               └── 技术选型
   │            │               │                   技术架构
   │            │               └── 数据建模（ER图）
   │            │                   流程与角色设计
   │            │                   界面设计
   │            │                   权限设计（RBAC）
   │            │                   埋点设计
   │            └── 核心业务流程
   │                产品定位
   │                应用架构
   │                功能模块
   │                演进蓝图
   └── 调研目标
       调研方法（访谈、观察、文档）
       问题总结
```

**核心原则**：自顶向下，从抽象到具体，先整体后细节。

## 2. B 端 vs C端建设差异

**来源**: Ch 3, p95-105 | **PM 相关度**: ⭐⭐⭐

| 维度 | B 端产品 | C 端产品 |
|------|---------|---------|
| **设计起点** | 业务调研（解决业务问题） | 商业分析（市场/用户群分析） |
| **MVP 思路** | 保证核心业务流程运转，系统有一定复杂度 | 聚焦核心痛点，可能只含一两个功能点 |
| **细节设计** | 关注建模、抽象、角色、权限 | 关注用户体验、交互设计 |
| **运营依赖** | 上线后公司要求使用，运营相对简单 | 上线只是第一步，高度依赖运营推广 |
| **需求来源** | 组织和机构（业务部门） | 个体用户 |
| **决策方式** | 集体决策（采购、审批、使用、管理） | 个人决策 |

**关键洞察**：B 端产品上线即有用户（内部强制使用），C 端产品上线只是运营的开始。

## 3. 业务调研方法论

**来源**: Ch 4, p95-113 | **PM 相关度**: ⭐⭐⭐

```
业务调研四步法

1. 明确调研目标
   └── 了解业务现状、发现业务问题、挖掘系统建设诉求

2. 制定调研计划
   └── 确定调研对象（业务负责人、一线人员、技术负责人）
       确定调研方式（访谈、观察、文档分析）

3. 执行调研
   └── 关键访谈问题：
       - 当前业务流程是怎样的？
       - 有哪些痛点和效率瓶颈？
       - 期望系统解决什么问题？
       - 未来业务发展方向是什么？

4. 总结调研结论
   └── 输出：业务现状描述 + 业务问题清单 + 需求优先级排序
```

**调研关键原则**：
- 邀请技术负责人参与调研，确保业务理解一致
- 通过业务调研找到**关键业务问题**，这是设计产品方案的核心前提
- 调研不仅是收集需求，更是理解业务本质的过程

## 4. 整体方案设计五步法

**来源**: Ch 5, p114-185 | **PM 相关度**: ⭐⭐⭐

```
整体方案设计（自顶向下）

Step 1: 核心业务流程
└── 梳理业务主干脉络，明确哪些环节需要线上化
    绘制跨部门核心业务流程图

Step 2: 产品定位
└── 产品针对谁提供什么支持
    明确子系统划分（如：商城前台 + 客户后台 + 运营后台）
    原则：独立业务部门 → 独立系统

Step 3: 应用架构设计
└── 新系统如何与现有系统融合
    复用 vs 新建 vs 改造的决策
    与架构师/CTO 共同确定

Step 4: 功能模块设计
└── 为每个系统设计功能模块（一二级导航结构）
    先做加法（穷举所有可能的功能）
    再做减法（按优先级筛选）

Step 5: 演进蓝图设计（Roadmap）
└── 根据业务优先级拆分实现节奏
    一期（MVP）→ 二期 → 三期 ...
    每期目标明确、边界清晰
```

**设计起点**：从核心业务流程切入，而非从功能点开始。

## 5. RBAC 权限模型

**来源**: Ch 6, p185-244 | **PM 相关度**: ⭐⭐⭐

```
RBAC（Role Based Access Control）权限模型

用户(User) → 角色(Role) → 权限组(Permission Group) → 资源(Resource)

┌──────────┐     ┌──────────┐     ┌──────────────┐     ┌──────────┐
│   用户    │──N:N──│   角色    │──N:N──│  权限组       │──N:N──│  资源    │
│ (User)   │     │ (Role)   │     │ (Permission) │     │(Resource)│
└──────────┘     └──────────┘     └──────────────┘     └──────────┘
                       │
                  ┌────▼────┐
                  │ 用户组   │
                  │(Group)  │
                  └─────────┘
```

**权限两大维度**：

| 维度 | 说明 | 示例 |
|------|------|------|
| **功能权限** | 角色→页面→按钮 | 管理员可增删改，运营只能查看 |
| **数据权限** | 角色→数据范围 | 分公司管理员看全部订单，区域主管只看本区域 |

**设计要点**：
- 权限设计的前提是合理的角色设计
- 按钮可见性控制 ≠ URL 访问控制，两者都要做
- 用权限表（Permission Matrix）描述角色-页面-操作的对应关系

## 6. 数据建模（业务建模）

**来源**: Ch 6, p185-244 | **PM 相关度**: ⭐⭐⭐

```
数据建模三步骤

1. 实体识别
   └── 从业务描述中提取名词 → 实体
       例：客户、门店、订单、商品、价格系数表

2. 关系定义
   └── 实体之间的关系（1:1, 1:N, N:N）
       例：一个客户拥有多个门店（1:N）
           一个门店有多个子账号（1:N）

3. ER 图绘制
   └── 用实体关系图表达业务数据模型
       这是产品设计的基石，保证系统的灵活性和扩展性
```

**核心观点**：数据建模是细节方案设计中最重要的环节，是保证产品设计严谨可行的关键工作。

## 7. B 端界面设计模式

**来源**: Ch 6, p185-244 | **PM 相关度**: ⭐⭐⭐

```
B端界面设计三要素

1. 页面流转设计
   └── 梳理用户在各页面间的跳转路径
       区分"导航可见页面"和"入口触发页面"

2. 页面详细设计
   └── 列表页：查询条件 + 表格 + 分页 + 操作按钮
       表单页：字段分组 + 校验规则 + 提交流程
       详情页：信息展示 + 关联操作 + 状态流转

3. 原型交互
   └── 提供可体验的交互原型
       让业务用户提前感受并反馈意见
       减少不必要的返工
```

**设计原则**：
- 功能模块图就是系统的骨架，具体功能是对骨架的填充
- 独立的业务部门应该由独立系统支持
- B 端界面重功能完整，C 端界面重体验流畅

## 8. 需求管理与迭代方法论

**来源**: Ch 10, p277-295 | **PM 相关度**: ⭐⭐⭐

### 需求管理

```
需求收集 → 初步判断 → 需求池管理 → 优先级排序 → 迭代计划

判断需求的四个问题：
1. 这个需求背后的真正问题是什么？
2. 这个问题是否有简单快速的解法？
3. 这个问题的影响面有多大？（个案 vs 共性）
4. 如果是共性问题，优先级和紧急程度如何？
```

**需求池管理模板关键字段**：业务线、需求类型、主题、内容、来源、提出日期、优先级、状态、负责人、预期上线日期、关联需求、开发评估工时、开发负责人、测试负责人、上线日期。

### 迭代管理

```
迭代管理要点
├── 迭代周期：根据公司节奏灵活设定（不一定是两周一迭代）
├── 计划外需求：标记"插入"类型，控制范围蔓延
├── 迭代目标：每期目标明确、边界清晰
└── 透明化：让所有相关方了解项目进度
```

**关键洞察**：C 端的 KANO 模型不适用于 B 端。B 端需求来自组织和机构，动机是解决业务问题。

## 9. B 端项目管理要点

**来源**: Ch 8, p245-265 | **PM 相关度**: ⭐⭐

```
B端项目管理核心

PMO / 产品负责人职责
├── 设计并优化项目管理制度
│   ├── 结合公司实际情况调整业界最佳实践
│   ├── 合理的规范应约束团队也保护团队
│   └── 管理项目管理工具（JIRA、Teambition）
│
└── 负责大中型项目的立项实施
    ├── 需求评审
    ├── 项目排期
    ├── 风险控制
    ├── 跨端协调（B端项目经常跨多个系统）
    └── 项目验收

关键原则：
- 作坊式 → 标准化：随公司规模增长必须升级管理模式
- 项目管理制度要"因司制宜"，不能照搬
- B端项目牵涉面广，跨端协调是最大难点
```

## 10. 企业级应用架构

**来源**: Ch 12-15, p311-369 | **PM 相关度**: ⭐⭐⭐

```
企业级应用架构 = 企业所有软件系统有机集成的方式

设计原则：
├── 分层设计：用户层 → 应用层 → 服务层 → 数据层
├── 组件化：保持系统的灵活性和扩展性
├── 服务化：通过 API 实现系统间松耦合
├── 主数据管理：解决信息孤岛问题
└── 按业务组织划分系统：独立业务部门 → 独立系统

架构演进规律（以 M 集团为例）：
单系统 → 多系统 → 平台化 → 服务化 → 中台化

应用架构图的含义：
├── 整体结构特征：系统如何划分和组织
├── 逻辑分层特征：各层职责和依赖关系
└── 功能模块抽象特征：通用能力的复用方式
```

**PM 学习架构的益处**：
1. 加深对业务和产品设计的理解
2. 把握架构级方案设计的能力
3. 拓展职业发展空间（产品总监/架构师方向）
