# About Face 4 - 核心模型与框架

> 从全书提炼的核心方法论、模型和思维框架。这些是交互设计领域最有价值的可迁移知识。

---

### Goal-Directed Design（目标导向设计）

**定义**：以用户目标为驱动力的完整产品设计方法论。流程包括六个阶段：Research（定性研究）→ Modeling（Persona 创建与目标分析）→ Requirements（场景叙事与需求定义）→ Framework（设计框架）→ Refinement（细节打磨）→ Validation（可用性测试）。核心理念是：产品行为应由用户目标决定，而非技术实现或业务逻辑。

**适用场景**：任何数字产品的用户体验设计规划。尤其适合从零开始设计新产品，或在现有产品中系统性地解决体验问题。是替代"功能堆砌"式产品规划的完整替代方案。

**关联章节**：[Ch01](./Ch01.md), [Ch02](./Ch02.md), [Ch03](./Ch03.md), [Ch04](./Ch04.md), [Ch05](./Ch05.md)

### Persona（人物画像）

**定义**：将定性研究数据浓缩为一个或多个具象化的"典型用户"原型——包含姓名、照片、人口统计、行为模式、目标和痛点。Persona 的核心价值是作为设计决策的"锚点"：当团队争论"用户需要什么"时，回到"如果 Beth 遇到这个情况会怎样"。Persona 是 Archetype（原型）而非 Stereotype（刻板印象），应基于真实用户数据构建。

**适用场景**：跨职能团队对齐"为谁设计"时。尤其在需求优先级讨论中，Persona 能有效避免"弹性用户"（Elastic User）和"自我参照设计"（Self-Referential Design）两大陷阱。

**关联章节**：[Ch03](./Ch03.md)

---

### 用户目标三层次（Three Levels of User Goals）

**定义**：用户目标分为三个递进层次——**Experience Goals**（体验目标）：用户希望获得的情感体验（如"感到掌控"）；**End Goals**（最终目标）：用户希望通过产品完成的具体任务（如"发送报告"）；**Life Goals**（人生目标）：用户的长期个人志向（如"成为优秀的项目经理"）。产品应优先满足 End Goals，因为它们是直接可操作的设计输入。

**适用场景**：在确定产品功能优先级时。当功能取舍产生分歧时，用目标层次来判断——如果某功能服务于 Persona 的 End Goal，就保留；如果只服务于 Experience Goal 而阻碍 End Goal，就删除。

**关联章节**：[Ch03](./Ch03.md)

### 产品姿态（Product Posture）

**定义**：产品根据使用频率和用户投入程度，可归类为三种姿态——**Sovereign**（主权型）：用户长时间连续使用（如 IDE、ERP），占用全部注意力，需要丰富功能和效率优化；**Transient**（临时型）：用户间歇性使用（如天气应用、ATM），每次使用时间短，需要清晰导航和即时可理解性；**Daemonic**（守护型）：后台持续运行无需用户干预（如杀毒软件、同步工具），需要安静、高效、可配置。

**适用场景**：产品定义和设计策略的起点。姿态决定了从信息架构到视觉设计的几乎所有决策——Sovereign 应用可以牺牲简洁换取功能密度，Transient 应用则必须极度简化。

**关联章节**：[Ch09](./Ch09.md)

### 永久中间用户原则（Perpetual Intermediates）

**定义**：大多数用户既不是新手也不是专家，而是处于中间水平——他们了解产品的核心功能，但不掌握所有高级特性。用户不会沿着"新手→专家"的线性路径前进，而是停留在中间水平。因此，界面应面向中间用户优化，同时为新手提供渐进式学习路径，为专家提供效率快捷方式。

**适用场景**：功能复杂的产品设计。当团队争论"应该简化还是增加功能"时，这个原则提供了清晰答案：两者都要——但以中间用户为核心，用 Inflecting the Interface（界面屈折）技术让不同水平的用户获得适合的体验。

**关联章节**：[Ch10](./Ch10.md)

### 数字礼仪三维度（Digital Etiquette）

**定义**：好的产品应像有教养的人一样对待用户——**Considerate**（体贴）：尊重用户的注意力，不强迫不必要的中断，提供清晰的上下文和反馈；**Smart**（社交产品）：让用户之间的协作顺畅自然，清晰呈现社交关系和状态。

**适用场景**：评估产品"行为品质"的检查清单。尤其适用于评估产品是否让用户感到被尊重和控制。

**关联章节**：[Ch08](./Ch08.md)

---

### 消除冗余操作（Eliminating Excise）

**定义**：用户任务分为两类——**Goal-Directed Tasks**（目标导向任务）：用户为达成目标而主动执行的操作；**Excise Tasks**（冗余操作）：用户为实现目标所必须经过但本身不创造价值的操作（如导航、配置、确认对话框）。Excise 是用户体验的头号敌人，应通过减少目的地、提供路标、提供概览、正确映射等策略系统性消除。

**适用场景**：流程优化和界面简化。尤其适用于识别和消除 Navigation Excise（导航冗余）和 Admin Excise（管理冗余），这两种最常见的冗余类型。

**关联章节**：[Ch12](./Ch12.md)

---

### Implementation Model vs. Mental Model vs. Represented Model

**定义**：三个关键模型的区分——**Implementation Model**（实现模型）：产品在技术层面的实际运作方式；**Mental Model**（心智模型）：用户基于生活经验对产品运作方式的理解；**Represented Model**（呈现模型）：产品通过界面呈现给用户的行为模式。优秀设计的核心是让 Represented Model 尽可能接近用户的 Mental Model，而非反映 Implementation Model。

**适用场景**：任何涉及技术抽象的设计决策。当工程师的直觉是"按系统架构组织界面"时，这个框架提醒设计师必须将系统逻辑翻译为用户可理解的语言。

**关联章节**：[Ch01](./Ch01.md)

---

### Idiom 优于 Metaphor（Building Idioms）

**定义**：界面交互模式分为三种——**Metaphor**（隐喻）：借用用户熟悉的现实世界类比（如"桌面"、"文件夹"），但受限于类比对象的原始特征；**Idiom**（惯用法）：产品领域内特有的交互模式（如"拖拽移动"、"右键菜单"），不依赖现实类比，可以自由创新；**Affordance**（示能）：通过视觉线索暗示操作可能性（如凸起按钮暗示可点击）。随着用户数字素养的提升，Idiom 应逐渐取代 Metaphor 成为主要的交互范式。

**适用场景**：选择交互模式时的决策框架。当团队争论"用现实隐喻还是创造新交互"时，优先构建 Idiom 而非受限于 Metaphor 的表达天花板。

**关联章节**：[Ch13](./Ch13.md)

---

### 编排与流畅（Orchestration and Flow）

**定义**：产品交互应像交响乐一样编排——所有交互元素和谐协作，用户进入 Flow 状态（心流），忘记工具的存在。编排的关键原则包括：和谐的交互（不突兀的状态变化）、合理的动效和过渡（为用户提供上下文连续性）、以及"毫不费力"的终极理想。打断 Flow 的最大杀手是 Excise 和意外错误。

**适用场景**：评估多步骤流程的连贯性。尤其适用于识别流程中的"摩擦点"——任何让用户被迫退出 Flow 状态的交互都是改进机会。

**关联章节**：[Ch11](./Ch11.md)
