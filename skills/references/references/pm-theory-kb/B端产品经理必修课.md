## 1. B 端产品定义与分类

**来源**: Ch 1, p20-31 | **PM 相关度**: ⭐⭐⭐

**B 端产品的核心特征**：

| 特征 | 说明 |
|------|------|
| **用户是企业/组织** | 而非个人消费者 |
| **解决业务问题** | 效率、管理、决策 |
| **多角色协作** | 采购、审批、使用、管理 |
| **付费意愿强** | 企业预算，关注 ROI |
| **学习成本容忍度高** | 功能完整比简单更重要 |
| **定制化需求多** | 不同企业流程差异大 |

**B 端 vs C 端关键差异**：

```
           B 端                    C 端
        ┌─────────────┐        ┌─────────────┐
  用户    │ 企业/组织   │   用户   │ 个人消费者   │
        ├─────────────┤        ├─────────────┤
  目标    │ 提升效率    │   目标   │ 满足需求    │
        ├─────────────┤        ├─────────────┤
  决策    │ 集体决策    │   决策   │ 个人决策    │
        ├─────────────┤        ├─────────────┤
  付费    │ 企业预算    │   付费   │ 个人消费    │
        ├─────────────┤        ├─────────────┤
  使用    │ 培训+习惯   │   使用   │ 即时上手    │
        └─────────────┘        └─────────────┘
```

## 3. 精益产品思维

**来源**: Ch 3, p35-42 | **PM 相关度**: ⭐⭐⭐

> 精益思想的核心：消除浪费，持续改进，以最少的资源创造最大的价值。

**精益思想在产品中的应用**：

| 精益原则 | 产品实践 |
|---------|---------|
| **消除浪费** | 不做无用功能，砍掉低 ROI 需求 |
| **最小可行产品** | MVP 快速验证，避免过度设计 |
| **持续交付** | 小步快跑，频繁迭代 |
| **价值流** | 从用户价值出发，反推产品流程 |
| **拉动式生产** | 需求驱动开发，而非预测驱动 |

**核心心智模式**：
- 看不见的角落更要重视（B 端的边界场景）
- 以终为始（从业务目标反推产品方案）
- 拥抱变化（需求变更是常态，不是异常）

## 4. 需求蛋模型

**来源**: Ch 4, p53-55 | **PM 相关度**: ⭐⭐⭐

```
         ┌───────────┐
         │   需求蛋   │
         │  (表层)    │
         │ "我要XX"   │
         └─────┬─────┘
               │ 深挖
        ┌──────▼──────┐
        │  业务目标    │
        │  (核心)    │
        │ "为什么需要" │
        └──────┬──────┘
               │ 拆解
        ┌──────▼──────┐
        │  产品方案    │
        │  (解决方案)  │
        └─────────────┘
```

**原则**：不要直接接"需求"，要像剥洋葱一样层层深入，找到真正的业务目标。

## 5. D × V × F > R 公式

**来源**: Ch 4, p55-56 | **PM 相关度**: ⭐⭐⭐

```
D × V × F > R

D = Design（设计可行性）
V = Value（业务价值）
F = Feasibility（技术可行性）
R = Resource（资源成本）

只有当设计可行性、业务价值、技术可行性的乘积
大于资源成本时，才应该做这个需求。
```

**决策矩阵**：

| 情况 | 判断 |
|------|------|
| V 高, R 低 | 优先做 |
| V 高, R 高 | 分阶段做 |
| V 低, R 低 | 视情况做 |
| V 低, R 高 | 不做 |

## 6. 需求管理公交模型

**来源**: Ch 4, p77-85 | **PM 相关度**: ⭐⭐⭐

```
需求池 = 公交线路

       ┌───────┐
       │ 终点站  │ ← 战略目标
       └───┬───┘
           │
    ┌──────▼──────┐
    │  各站点     │ ← 各阶段里程碑
    │ (需求池)    │
    └──────┬──────┘
           │
    ┌──────▼──────┐
    │  乘客上下车 │ ← 需求进入/退出
    └─────────────┘
```

**核心规则**：
- 每个需求都有明确的优先级和重要性
- 需求按优先级排队（先到终点站的先到）
- 支持急诊模式（紧急需求插队）
- 定期清理无效需求（乘客到站下车）

## 7. 信息架构设计方法

**来源**: Ch 5, p95-107 | **PM 相关度**: ⭐⭐⭐

### 两步法

```
1. 解析信息架构（收纳信息）
   - 卡片分类法：将功能点写在卡片上，归类整理
   - 用户故事法：基于角色-场景-目标梳理功能
   - 业务流程法：按业务流程串联功能

2. 输出站点地图（原型设计的起点）
   - 层级结构：模块 → 页面 → 功能
   - 标注角色权限
   - 标注页面间跳转关系
```

### B 端信息架构特点

| 特点 | 说明 |
|------|------|
| **角色驱动** | 不同角色看到不同功能 |
| **流程导向** | 按业务流程组织，而非按功能 |
| **权限复杂** | 操作权限、数据权限、字段权限 |
| **集成需求** | 与其他系统对接 |

## 8. 原型设计模式思维

**来源**: Ch 5, p107-120 | **PM 相关度**: ⭐⭐⭐

**模式思维**：不要每次从零开始设计，积累可复用的设计模式。

```
设计模式库
├── 列表页模式（查询条件 + 表格 + 分页 + 操作）
├── 表单页模式（字段分组 + 校验规则 + 提交流程）
├── 详情页模式（信息展示 + 关联操作 + 状态流转）
├── 审批流模式（发起 → 审批 → 驳回 → 通过）
├── 权限配置模式（角色 → 菜单 → 按钮 → 数据）
└── 仪表盘模式（KPI 卡片 + 图表 + 趋势）
```

**登机模式**：从全局导航到具体页面的逐层细化方法。

## 9. KSF 方法（关键成功因素法）

**来源**: Ch 8, p143-147 | **PM 相关度**: ⭐⭐

```
KSF = Key Success Factors

步骤：
1. 明确产品目标
2. 分解影响目标的关键因素
3. 为每个因素设定可量化的指标
4. 设定目标值（基于历史数据 + 业务预期）
5. 监控偏差，及时调整
```

**数据指标的二律背反**：过度关注指标可能导致行为扭曲（Goodhart 定律）。

## 10. 绊倒 PM 成长的六条绳索

**来源**: Ch 12, p162-164 | **PM 相关度**: ⭐⭐

| 绳索 | 问题 | 解法 |
|------|------|------|
| **不懂业务** | 设计脱离实际 | 深入一线，理解业务流程 |
| **需求无序** | 被需求牵着走 | 建立需求管理机制 |
| **沟通低效** | 重复沟通、理解偏差 | 结构化沟通（文档+原型） |
| **技术盲区** | 无法评估可行性 | 学习基础技术知识 |
| **缺乏数据感** | 决策靠拍脑袋 | 掌握数据分析方法 |
| **成长停滞** | 重复工作无突破 | 精益改进 + 刻意练习 |
