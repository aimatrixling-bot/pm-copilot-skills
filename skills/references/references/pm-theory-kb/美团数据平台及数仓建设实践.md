## 1. 数仓分层架构

**来源**: Ch 1-3 | **PM 相关度**: ⭐⭐⭐

```
数仓分层模型（美团标准）

ODS (Operational Data Store) — 操作数据层
├── 原始数据，不做加工
└── 与业务库保持一致

DWD (Data Warehouse Detail) — 明细数据层
├── 清洗、标准化、维度退化
├── 统一数据口径
└── 采用维度建模（星型/雪花模型）

DWS (Data Warehouse Summary) — 汇总数据层
├── 按主题域汇总
├── 通用指标计算
└── 宽表设计

ADS (Application Data Store) — 应用数据层
├── 面向具体应用场景
├── 个性化指标
└── 直接对接报表/产品

DIM — 维度表
├── 缓慢变化维（SCD）
└── 维度退化
```

**PM 对应**：理解数仓分层是数据产品经理的基本功，决定了数据从哪里来、怎么加工、在哪里消费。

## 2. OneData 方法论

**来源**: Ch 2 | **PM 相关度**: ⭐⭐⭐

```
OneData 核心三统一

1. 统一业务归口（规范定义）
   └── 统一指标定义、统一维度定义
       消除同名不同义、同义不同名

2. 统一应用归口（规范生产）
   └── 统一数据出口
       避免重复计算、口径不一致

3. 统一数据出口（规范消费）
   └── 统一数据服务
       一份数据多处使用
```

**实施策略**：
- 自上而下：先梳理业务过程，再定义指标
- 自下而上：先盘点现有数据资产，再逐步规范化
- 迭代推进：优先治理高频使用的指标

**PM 对应**：OneData 是数据中台建设的核心方法论，直接决定指标体系的质量。

## 3. 数据治理体系

**来源**: Ch 10-11 | **PM 相关度**: ⭐⭐⭐

```
数据治理四大支柱

1. 组织保障
   └── 成立数据治理委员会
       明确各角色职责（数据Owner、数据Steward、数据消费者）

2. 标准规范
   ├── 命名规范（表名、字段名、指标名）
   ├── 开发规范（ETL流程、数据质量检查）
   └── 安全规范（权限管理、数据脱敏）

3. 技术平台
   ├── 数据地图（Wherehows）：血缘分析、影响分析
   ├── 数据质量平台（DataMan）：自动检核、报警、修复
   └── 元数据管理：统一元数据中心

4. 持续运营
   ├── 治理指标衡量（覆盖率、及时率、准确率）
   ├── 定期审计
   └── 培训推广
```

## 4. 数据质量检核框架

**来源**: Ch 12 | **PM 相关度**: ⭐⭐⭐

```
数据质量检核五维度

┌─────────────────────────────────────────────┐
│ 数据质量                                      │
├──────────┬──────────┬──────────┬──────────────┤
│ 完整性    │ 准确性    │ 一致性    │ 时效性        │
│ Completeness │ Accuracy │ Consistency │ Timeliness │
│          │          │          ├──────────────┤
│          │          │          │ 唯一性        │
│          │          │          │ Uniqueness   │
└──────────┴──────────┴──────────┴──────────────┘

检核方式：
├── 强规则：必须通过（如主键不为空）
├── 弱规则：允许一定比例异常（如枚举值检查）
└── 统计规则：基于历史分布的异常检测
```

**管理流程**：发现→登记→分配→修复→验证→关闭。

## 5. A/B 评估体系

**来源**: Ch 16 | **PM 相关度**: ⭐⭐⭐

```
A/B 评估体系三要素

1. 权威完备的指标体系
   ├── 北极星指标：唯一核心指标
   ├── 过程指标：拆解到可操作环节
   └── 护栏指标：确保不损害底线

2. 科学权威的评估方式
   ├── 分流策略：随机分流、分层分流
   ├── 实验周期：足够的样本量
   ├── 显著性检验：p-value、置信区间
   └── 辛普森悖论防范：分组分析

3. 评估流程
   ├── 实验前：假设设定、指标选择、样本量计算
   ├── 实验中：监控数据质量、提前终止条件
   └── 实验后：效果评估、决策建议
```

## 6. 实时数仓架构

**来源**: Ch 4-5 | **PM 相关度**: ⭐⭐

```
实时数仓架构（流批结合）

实时数据流：
Kafka → Flink(实时计算) → 实时OLAP(Doris/ClickHouse) → 实时大屏

离线数据流：
DB → Binlog → Kafka → HDFS → Hive/Spark → 离线数仓 → 定时报表

流批结合：
├── Lambda架构：实时层+离线层并行，结果合并
├── Kappa架构：纯流式，用消息队列保留全量数据
└── 流批一体：Flink统一引擎，API层面统一

SLA建设：
├── 延迟监控：端到端延迟 < 阈值
├── 可用性：99.9%+
└── 数据一致性：实时与离线结果对账
```

## 7. 数据产品化方法论

**来源**: Ch 8-9 | **PM 相关度**: ⭐⭐⭐

```
数据产品化四步法

Step 1: 识别数据需求
└── 从业务痛点出发，识别可被数据解决的问题
    与业务方深度沟通，理解决策场景

Step 2: 数据产品设计
├── 确定核心指标（少而精）
├── 设计分析维度（多角度下钻）
├── 设计交互流程（从总览到明细）
└── 设计可视化方案（图表选型）

Step 3: 数据生产
├── 指标口径统一（OneData）
├── 数据质量保障（自动化检核）
└── 更新频率确定（T+1/实时）

Step 4: 持续运营
├── 收集用户反馈
├── 优化指标和维度
├── 拓展使用场景
└── 建立数据产品运营机制
```

**PM 对应**：这是数据产品经理最核心的工作方法论。

## 8. OLAP 引擎选型

**来源**: Ch 14-15 | **PM 相关度**: ⭐⭐

| 引擎 | 模式 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| **Apache Kylin** | MOLAP（预计算） | 查询速度快（亚秒级） | 构建时间长、维度爆炸 | 固定维度、高并发查询 |
| **Apache Doris** | MOLAP+ROLAP（混合） | 实时导入、支持灵活查询 | 大规模场景资源消耗高 | 准实时、多维分析 |
| **ClickHouse** | ROLAP（列式存储） | 单表查询极快、压缩比高 | JOIN 性能弱 | 宽表分析、日志分析 |
| **Presto/Trino** | MPP（联邦查询） | 跨数据源查询 | 内存消耗大、不适合大数据量 | 即席查询、数据探查 |

## 9. 数据平台融合

**来源**: Ch 6 | **PM 相关度**: ⭐⭐

```
数据平台融合的四大难点

1. 架构复杂：多套系统、多种技术栈
2. 可靠性要求：线上业务依赖
3. 数据体量：PB级数据迁移
4. 平台化与复杂度：功能完整 vs 易用性

融合策略：
├── 数据互访打通：统一元数据、统一权限
├── 统一计算引擎：减少引擎数量
├── 统一存储格式：Parquet/ORC
└── 渐进式迁移：按业务线逐步切换
```

## 10. DB 数据同步架构

**来源**: Ch 13 | **PM 相关度**: ⭐⭐

```
DB → 数据仓库同步方案

Binlog实时采集 → Kafka → HDFS → Hive

关键组件：
├── Canal/Databus：Binlog采集
├── Kafka：消息缓冲
├── Camus：Kafka2HDFS（按时间分区）
├── Merge：增量合并到全量
└── Checkdone：任务完成检测

实践要点：
├── 分库分表支持：合并多个数据源
├── 删除事件处理：标记删除而非物理删除
└── Schema变更同步：DDL事件处理
```
