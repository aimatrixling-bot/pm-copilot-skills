## 1. 妈妈测试三大规则

**来源**: Ch 1, p8-22 | **PM 相关度**: ⭐⭐⭐

```
妈妈测试三大规则

Rule 1: 不要问人们想要什么
└── 人们会撒谎（不是故意的，是为了礼貌、面子、讨好你）
    "这主意太棒了！" ≠ "我会买的"

Rule 2: 只问关于过去和现在的事实
└── 事实 ≠ 观点/预测/想法
    ✓ "你上次解决X问题时是怎么做的？"
    ✗ "你觉得X功能好不好？"

Rule 3: 谈论具体的，而非抽象的
└── 聚焦真实行为，而非假设场景
    ✓ "给我讲讲你上一次遇到这个问题的经历"
    ✗ "如果有一个产品能解决X，你会用吗？"

核心原则：
├── 客户访谈是"考古挖掘"，不是"推销产品"
├── 你的妈妈永远会说你的主意很好 → 所以叫"妈妈测试"
└── 目标：发现真实问题和真实行为，而非验证你的想法
```

## 2. 好问题 vs 坏问题

**来源**: Ch 1, p14-21 | **PM 相关度**: ⭐⭐⭐

```
坏问题（会让你得到假数据）：

| 坏问题 | 为什么坏 | 好问题替代 |
|--------|---------|-----------|
| "你觉得这主意好吗？" | 寻求认可，不是事实 | "你目前怎么处理X？" |
| "你会买能做X的产品吗？" | 假设场景，答案不可靠 | "你上次买X类产品是什么时候？" |
| "你愿意付多少钱？" | 口头意愿≠实际行为 | "你上次在这上面花了多少钱？" |
| "你梦想中的产品是什么样的？" | 太抽象，无法行动 | "你最近用了什么替代方案？" |
| "你为什么不用X？" | 让人辩护或编理由 | "告诉我上一次X不管用的经历" |

好问题（能得到真数据）：

| 好问题 | 为什么好 | 发现什么 |
|--------|---------|---------|
| "给我讲讲上次发生这事的经历" | 具体事实 | 问题的真实严重程度 |
| "你还试过什么其他方法？" | 现有解决方案 | 竞品和市场认知 |
| "你现在怎么处理这个问题？" | 当前行为 | 需求的真实性 |
| "钱从哪里来？" | 购买力 | 支付意愿的真实约束 |
| "我还能和谁聊这个？" | 推荐更多用户 | 扩大样本 |
| "有没有什么我应该问但没问的？" | 意外发现 | 你没想到的问题 |
```

## 3. 三种坏数据

**来源**: Ch 2, p23-40 | **PM 相关度**: ⭐⭐⭐

```
三种坏数据

┌────────────────────────────────────────────────┐
│  1. 赞美 (Compliments)                          │
│  ├── "这主意太棒了！""加油！"                     │
│  ├── 本质：社交润滑剂，不是需求信号               │
│  └── 应对：微笑 → 转移话题 → 回到事实问题         │
├────────────────────────────────────────────────┤
│  2. 废话 (Fluff)                                │
│  ├── "市场很大""大家都需要这个"                   │
│  ├── 本质：听起来像洞察，实际上什么都没说          │
│  └── 应对：锚定到具体数字/时间/事件               │
├────────────────────────────────────────────────┤
│  3. 想法 (Ideas)                                │
│  ├── "你应该加个X功能""如果能做Y就好了"            │
│  ├── 本质：解决方案，不是问题                     │
│  └── 应对：挖掘想法背后的动机                     │
│      "是什么让你觉得需要X？"                     │
└────────────────────────────────────────────────┘

应对坏数据的六个技术：
├── 转移赞美 (Deflect compliments)
├── 锚定废话 (Anchor fluff)
├── 挖掘想法之下 (Dig beneath ideas)
├── 停止寻求认可 (Stop seeking approval)
├── 切断推销 (Cut off pitches)
└── 少说话 (Talk less)
```

## 4. 对话货币模型

**来源**: Ch 5, p62-73 | **PM 相关度**: ⭐⭐

```
对话三种货币

会议成功 = 获得进展（Advancement）
进展 = 客户付出了某种货币

┌────────────────────────────────────────────────┐
│  货币              │  信号                        │
├───────────────────┼─────────────────────────────┤
│  时间 (Time)       │  "下次什么时候再聊？"          │
│  时间是最低级货币  │  "我们约下周二吧"              │
│                   │  "我能再花30分钟吗？"           │
├───────────────────┼─────────────────────────────┤
│  声誉 (Reputation) │  "我可以介绍几个人给你"        │
│  中级货币          │  "我们在公司内部讨论一下"       │
│                   │  "我帮你在LinkedIn上推荐"      │
├───────────────────┼─────────────────────────────┤
│  金钱 (Money)     │  "原型能卖给我吗？"            │
│  最高级货币        │  "什么时候能开始试用？"          │
│                   │  "我们可以预付定金"             │
└───────────────────┴─────────────────────────────┘

坏会议信号（没有获得进展）：
├── "太酷了！继续加油！" → 礼貌性的终结
├── "做好了告诉我" → 不愿意投入任何货币
└── "我会买的" → 口头承诺，没有实际行动
```

## 5. 客户切片（Customer Slicing）

**来源**: Ch 7, p89-98 | **PM 相关度**: ⭐⭐

```
客户切片四步法

Step 1: 定义问题（Who）
├── 谁有这个问题？
└── 他们必须是什么样的？

Step 2: 定义情境（Where）
├── 在什么情境下出现？
└── 何时何地发生？

Step 3: 配对（Who + Where）
├── 不是所有"Who"都有同样的问题
├── 不是所有"Where"都同样严重
└── 找到 Who × Where 的最佳组合

Step 4: 验证
├── 和切片后的客户交谈
└── 确认问题真实存在且有付费意愿

关键洞察：
├── "婴儿还是健美者？"——不要试图同时服务所有人
├── "大品牌还是夫妻店？"——客户规模决定产品策略
├── 和错误的人交谈 = 浪费时间
└── 早期应该聚焦于最有痛点的最小客户群体
```

## 6. 用户访谈流程

**来源**: Ch 8, p99-113 | **PM 相关度**: ⭐⭐

```
Prep → Meet → Review 三段流程

Prep（会前准备）：
├── 准备 3 个要验证的假设
├── 准备具体的好问题
├── 不要准备演示/推销
└── 目标：学习，不是销售

Meet（会议中）：
├── 保持随意（不是正式采访）
├── 遵循 3 个规则 + 避开坏数据
├── 追求进展（获取货币）
└── 20-30 分钟即可

Review（会后回顾）：
├── 24 小时内整理笔记
├── 提炼事实和学到的信息
├── 更新假设（验证/推翻/修正）
└── 记录跟进行动

笔记记录原则：
├── 记录事实，不记录观点
├── 记录原话（引用）
├── 标注来源（谁说的）
└── 固定格式，便于后续分析
```

## 7. 找到对话机会

**来源**: Ch 6, p74-88 | **PM 相关度**: ⭐⭐

```
获取客户对话的三种路径

路径 1：主动找他们（Going to them）
├── 冷启动电话/邮件
├── 偶遇（咖啡店、行业活动）
├── 找个好借口（请求建议而非推销）
├── 沉浸到客户所在环境
└── 落地页测试

路径 2：吸引他们来（Bringing them to you）
├── 组织 meetup
├── 演讲/教学
├── 行业博客
└── 创意方法

路径 3：温暖介绍（Warm intros）
├── 7 度人脉理论
├── 行业顾问
├── 大学
├── 投资人
└── 还人情

邀约框架（VFWPA）：
├── Vibe：轻松的语气
├── Frame：请求建议，不是推销
├── Win：强调对他们的价值
├── Praise：认可他们的专业
├── Ask：明确的请求
```
