# PM Copilot 共享质量机制

> 本文件被 Agent CLAUDE.md 和各 Skill 引用，定义跨所有 Skill 共享的质量增强机制。

## 零、保真度分级协议

本文件定义的检查机制按保真度分级执行：

| 段落 | Draft | Review（默认） | Release |
| --- | --- | --- | --- |
| 一、一致性检查 | 仅 User Journey ⊆ In Scope | 全部 4 条 | 全部 4 条 |
| 二、可行性挑战 | 不触发 | 命中触发词时触发 | 命中触发词时触发 |
| 三、风险缓解格式 | 不强制 4 字段 | 强制 4 字段 | 强制 4 字段 + 要求 ≥1 个边缘场景 |
| 四、反合理化 | 不检查 | 全部 | 全部 |
| 五、Loophole Detection | 不触发 | 不触发 | 强制执行 |
| 六、反向提问 | 1 个 | 2 个 | 3 个 |

用户显式声明 `fidelity` 标签时按标签执行；未声明时默认 Review。推断规则见 Agent CLAUDE.md Iron Law 约束段落。

## 零点五、对话输出格式协议（所有保真度均执行）

Skill 在**对话中**直接呈现给用户的文本产出必须遵守以下规则：

- [ ] **格式匹配产出类型** — 文档类（PRD/竞品分析/决策记录等）用纯 Markdown；原型/代码类按技术栈输出（HTML/React/TS）；文件导出类（docx/pdf/pptx）由对应 Skill 按目标格式处理
- [ ] **对话文本禁止非渲染标签** — 对话中的文本产出不得包含 `<generative-ui-widget>`、`<style>`、`<script>` 等无法在 Markdown 渲染中正确显示的标签
- [ ] **文档类产出不包裹代码围栏** — PRD 等文档内容直接输出为 Markdown，不放在 ```` ```markdown ``` ```` 中（代码片段本身除外）
- [ ] **Agent 说明与正文分离** — Agent 的解释性说明（意图解读、执行路径、自检备注）放在产出物之前或之后，与 Skill 模板产出物明确分隔

## 一、一致性检查协议

当交付物包含多个部分（如 Problem Statement + User Journey + Scope + Metrics + Risks）时，交付前必须验证：

- [ ] **User Journey ⊆ In Scope** — 用户旅程/使用流程中每个步骤都在 In Scope 中有对应功能，Out of Scope 功能不出现在 MVP 旅程中
- [ ] **功能引用自洽** — 所有功能引用 ⊆ 当前文档定义的功能集（不引用未定义的功能）
- [ ] **指标可追溯** — 所有成功指标引用都有对应的基线 + 目标 + 测量方式
- [ ] **风险缓解可执行** — 每个风险项有具体缓解措施（不是"持续监控"、"关注"等）

违反任一条 = 立即修正后才能交付。

## 二、可行性挑战触发词表

当输出中包含以下模式词时，**必须触发【技术实现猜想】**：

| 触发词 | 风险级别 | 要求的动作 |
| --- | --- | --- |
| 跨平台 / 多平台聚合 / 多平台同步 | 高 | 必须命名具体的 API / 协议 / 数据源，不是抽象的"集成" |
| 批量发送 / 批量操作 | 高 | 必须解释批处理机制、限流策略、失败重试 |
| 自动追踪 / 自动同步 / 自动更新 | 高 | 必须指定数据源和轮询/webhook 机制 |
| 实时 / 即时 | 中 | 必须定义延迟预算和可接受范围（如 P95 < 3s） |
| 一键部署 / 一键发送 / 一键XX | 中 | 必须列出隐藏的前提条件 |
| AI 自动 / 智能 XX | 中 | 必须定义模型输入/输出和降级方案 |
| 无缝集成 | 中 | 必须命名集成点（API 端点、webhook URL、SDK 方法名） |
| 全平台 / 所有 XX | 低 | 必须明确列举包含和不包含哪些平台/项目 |
| 第三方 / 外部系统 | 低 | 必须指定集成类型（官方 API / 爬虫 / RPA / 邮件 / 手动） |

### 【技术实现猜想】格式

对每个命中的触发词，在输出中以标注块形式补充：

```
**[触发]** _命中的触发词_
**[路径]** 具体技术路径（不是"使用 API"而是"调用供应商 REST API v2 /orders/batch 端点，限流 100/min"）
**[失败回退]** 如果此路径不可行，最简替代方案是什么？
**[最小可行方案]** 去掉此功能的今天就能发布的版本是什么？
```

如果用户说"不需要可行性分析"，可以跳过但必须标注 `[skip-feasibility-check: 用户确认]`。

## 三、风险缓解强制格式

每个风险项**必须**包含以下 4 个字段：

| 字段 | 要求 | 差的示例 | 好的示例 |
| --- | --- | --- | --- |
| **风险** | 具体场景，不是类别 | "性能风险" | "搜索响应时间 > 2s，当商品目录超过 1 万件时" |
| **缓解** | 具体动作，不是"监控" | "持续监控系统" | "预计算 Top 100 查询的搜索索引；长尾查询显示加载动画" |
| **指标** | 可衡量的护栏 | — | "P95 搜索延迟 < 1.5s；如果 P95 > 3s → 显示降级结果横幅" |
| **负责人** | 谁来执行 | "团队" | "后端工程师 + SRE 发布周值班" |

缓解方案写"持续监控"或"关注"等同于没有缓解。必须替换为具体动作。

## 四、反合理化补充表

以下借口在可行性检查时特别容易出现，必须警惕：

| 你可能在想的 | 真相 |
| --- | --- |
| "这个集成很标准" | "标准集成"通常忽略了认证、限流、错误处理、版本兼容 |
| "技术细节可以之后再补" | 没有《技术实现猜想》的需求是愿望，不是需求 |
| "这个风险已经在风险区了" | 列出风险 ≠ 缓解风险。没有具体缓解的风险项是占位符，不是保障 |
| "用户旅程是高层的，不需要和 Scope 匹配" | 每个用户旅程步骤都隐含对功能的承诺。Out of Scope 的步骤 = 未承诺的承诺 |
| "AI 会处理边缘情况" | "AI"不是技术方案。哪个模型？什么输入？什么降级？什么准确率阈值？ |
| "这个场景不太可能发生" | "不太可能" ≠ "不会发生"。未定义的行为 = 未定义的用户体验 |

## 五、Loophole Detection（仅 Release 保真度触发）

当保真度为 Release 时，在一致性检查之后额外执行漏洞检测。

### 触发条件

交付物中包含以下任一元素时触发：

- 状态流转（实体有 ≥ 2 种状态）
- 多角色协作（≥ 2 种角色参与同一流程）
- 时间依赖（流程中有先后顺序的时间节点）
- 资源所有权转移（数据/客户/任务在不同角色间流转）

### 检测方法

对每个触发的元素，逐一检查：

1. **状态转换完整性** — 每个状态是否有明确的前置状态和后续状态？是否存在"只能进不能出"的状态？
2. **角色交接定义** — 资源/任务从角色 A 转到角色 B 时，交接条件是否明确？交接失败如何处理？
3. **时间窗口边界** — 有时间限制的流程，超时后如何处理？提前完成是否有影响？
4. **并发冲突** — 两个角色同时操作同一资源时，冲突如何解决？
5. **回退路径** — 流程走到某步需要回退时，是否定义了回退规则？

### 输出格式

检测到漏洞时，在交付物中追加：

```markdown
## 漏洞检测结果

| # | 场景 | 未定义的行为 | 建议补充规则 | 优先级 |
| --- | --- | --- | --- | --- |
| 1 | {场景描述} | {当前文档未覆盖的行为} | {建议的最小规则定义} | P0/P1 |
```

检测到 0 个漏洞时，输出：`**漏洞检测**: 未检测到逻辑漏洞（基于当前文档定义范围）。`

## 六、反向提问机制（所有保真度均适用）

交付物末尾的"下一步"段落中，自动追加反向提问，引导用户思考。

### 格式

```markdown
## 反向提问

> 交付物已完成。在继续之前，考虑以下问题：

1. **{关键假设验证}**: 如果 [{核心假设}] 不成立，这份文档中最先需要改的是哪个部分？
2. **{下一步优先级}**: 下一步建议执行 {Skill 名}，但你是否更需要先 [{替代动作}]？
3. **{未覆盖的视角}**: 这份文档是否需要 [{不同受众}] 的视角补充？
```

### 选择规则

- 提问 1：来自交付物中置信度最低的假设
- 提问 2：推荐当前阶段的下一个 Skill，但提供替代选项
- 提问 3：基于交付物受众的盲点（如 PRD 是否考虑了技术视角、竞品分析是否考虑了用户视角）

提问数量：Draft 1 个，Review 2 个，Release 3 个。

## 七、用户不满时的诊断引导

当用户对产出表达不满时（触发信号："不对"、"不是这个意思"、"重来"、"没用"、"搞错了"），不要直接重试。先引导用户定位根因。

### 流程

1. 识别不满信号
2. 判断属于哪类根因（参考 `references/pm-debug-guide.md`）：
   - Context 缺失（AI 拿不到判断所需的输入）
   - Context 饱和（关键信息被稀释）
   - 指令歧义（请求有多种合理理解）
   - 能力边界（任务超出当前稳定能力）
3. 针对性修复（补信息 / 精简输入 / 改写目标 / 换策略）

### 与反向提问的关系

- 反向提问在**交付后**触发，预防下一个决策失误
- 诊断引导在**不满时**触发，修复当前产出问题
- 两者互补，不冲突
