# PM Debug Guide

当用户对 AI 产出表达不满时，帮助定位根因，而非盲目重试。

## 四类根因 × PM 场景

### 1. Context 缺失 → Hallucination

AI 编造了不存在的事实、引用了不相关的竞品、产出了脱离业务背景的内容。

| 症状 | 诊断问题 | 用户动作 |
|------|---------|---------|
| PRD 写了不存在的用户需求 | "你给 AI 看了什么用户数据？ persona 定义了吗？" | 运行 `/pm-persona` 定义用户画像后再写 PRD |
| 竞品分析选了不相关的竞品 | "你告诉 AI 你的产品定位和目标用户了吗？" | 先明确产品定位，再运行 `/pm-comp` |
| 原型使用了错误的交互模式 | "你提供了设计参考或现有界面截图吗？" | 提供截图或 wireframe 作为输入 |

### 2. Context 饱和 → 信息丢失

AI 忽略了你明确给过的约束，前后矛盾，重复已解决的问题。

| 症状 | 诊断问题 | 用户动作 |
|------|---------|---------|
| PRD 功能范围超出了你说的边界 | "你是不是把整个项目文档都贴进去了？" | 只提供当前任务相关的信息，用 `/pm-decision` 明确决策边界 |
| 竞品分析遗漏了你强调过的维度 | "信息太多时，AI 会稀释关键指令。精简输入试试。" | 用 Quick Entry Mode，明确列出分析维度 |
| 路线图忽略了已确认的约束 | "对话太长了，早期约束可能被遗忘。" | `/clear` 开新 session，只带入关键约束 |

### 3. 指令歧义 → 方向偏差

AI 做了"合理的"事但不是你想要的。

| 症状 | 诊断问题 | 用户动作 |
|------|---------|---------|
| "帮我写个 PRD" 产出了新功能 PRD，但你要的是现有功能迭代 | "'写 PRD' 可以有很多理解——是全新功能还是现有功能迭代？" | 用明确的范围描述："写现有预约功能的迭代 PRD，增加日内改期能力" |
| 竞品分析做了全面扫描，但你要的是特定维度对比 | "你需要全面竞品分析还是特定维度的对标？" | 明确分析目的："对比 3 个竞品的定价策略" |
| 原型做了全新设计，但你要的是局部优化 | "你描述的是整体重设计还是局部改进？" | 提供 before/after 对比参考 |

### 4. 能力边界 → 质量天花板

多次修改后仍不稳定，复杂场景反复出错。

| 症状 | 诊断问题 | 用户动作 |
|------|---------|---------|
| 竞品分析缺乏最新数据 | "这个分析是否需要 AI 无法获取的最新市场数据？" | 运行 `/pm-comp` 后，用 Ultra-research 补充实时数据 |
| 原型交互过于复杂，反复出错 | "这个交互是否可以拆分为更简单的步骤？" | 拆分为多个简单组件，分步实现 |
| 技术可行性判断不准确 | "PM 视角的技术评估是否需要工程师确认？" | 在 PRD 中标注 `[[待技术验证]]`，降低保真度到 Draft |

## 升级协议

同类型问题出现 3 次后，输出结构化总结：
1. 已尝试的修复方式（至少 2 种）
2. 判断的根因类型和依据
3. 建议的替代路径（换 Skill / 拆任务 / 降级保真度 / 补充输入）

## 与 Skill 的联动

| 根因类型 | 推荐的解决 Skill |
|---------|----------------|
| Context 缺失 | `/pm-persona`（补用户画像）、`/pm-comp`（补竞品数据） |
| Context 饱和 | `/pm-decision`（明确决策边界）、`/clear`（开新 session） |
| 指令歧义 | 重新描述时使用 Quick/Expert Entry Mode 的确认流程 |
| 能力边界 | `/pm-experiment`（设计验证替代深度分析）、降级到 Draft 保真度 |
