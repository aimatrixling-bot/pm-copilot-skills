# 用户测试方法指南

> 从测试计划到结果分析的完整流程。

---

## 1. 测试类型

| 类型 | 用途 | 样本量 | 耗时 |
|------|------|--------|------|
| **可用性测试** | 发现 UX 问题 | 5-8 人 | 1-2 周 |
| **A/B 测试** | 量化比较方案 | ≥1000/变体 | 2-4 周 |
| **探索性测试** | 发现未知问题 | 3-5 人 | 2-3 天 |
| **回归测试** | 验证修复 | 按用例数 | 1-3 天 |
| **冒烟测试** | 基本功能验证 | 自动化 | 几小时 |

---

## 2. 可用性测试流程

### 招募参与者

- **数量**: 5 人可发现 80% 可用性问题（Jakob Nielsen 法则）
- **标准**: 目标用户画像匹配
- **多样性**: 经验水平混合（新手+高级）

### 测试会议流程（标准 60 分钟）

| 时间 | 环节 | 活动 |
|------|------|------|
| 5 min | 开场 | 说明目的、流程、权利 |
| 5 min | 前测问卷 | 收集背景信息 |
| 30-40 min | 任务执行 | 观察用户行为，不引导 |
| 10 min | 访谈 | 收集反馈和感受 |
| 5 min | 结束 | 感谢和下一步 |

### 关键原则

1. **观察而非引导** — 让用户自己解决问题
2. **记录一切** — 行为、表情、语言
3. **"出声思考"法** — 让用户边做边说想法
4. **不帮用户** — 看到用户困惑时忍住不提示

## 3. 测试用例设计

### 好的测试任务

```
好的: "你想要查看上个月的订单，请找到相关信息"
坏的: "点击右上角的'订单'按钮"（引导了操作）
```

### 测试用例结构

| 字段 | 说明 |
|------|------|
| User Story | 对应哪个用户故事 |
| 测试项 | 测试什么功能 |
| 前置条件 | 开始前需要什么 |
| 测试步骤 | 用户要做什么 |
| 预期结果 | 成功应该看到什么 |
| 优先级 | P0/P1/P2 |
| 类型 | 正向/边界/异常 |

### 测试覆盖策略

| 优先级 | 覆盖范围 | 测试类型 |
|--------|----------|----------|
| P0 | 100% 覆盖 | 手动 + 自动 |
| P1 | 核心路径 | 手动为主 |
| P2 | 探索性抽检 | 探索性 |

## 4. 结果分析

### 可用性问题评级

| 级别 | 标准 | 行动 |
|------|------|------|
| **严重** | 用户无法完成任务 | 必须修复 |
| **中等** | 用户完成但遇到明显困难 | 应该修复 |
| **轻微** | 小的摩擦点 | 可以优化 |

### SUS 评分（系统可用性量表）

| 分数 | 评级 |
|------|------|
| 90-100 | 最佳 |
| 80-89 | 优秀 |
| 70-79 | 良好 |
| 60-69 | 一般 |
| 50-59 | 差 |
| < 50 | 极差 |

---

## 5. 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 引导用户 | 得到虚假正面结果 | 只观察，不提示 |
| 样本太大 | 浪费资源 | 5 人足够发现 80% 问题 |
| 只测正常流程 | 遗漏边界情况 | 包含异常和边界测试 |
| 不记录过程 | 事后无法分析 | 录屏、记笔记、拍照片 |
| 测试太晚 | 发现问题来不及改 | 尽早测试（原型阶段） |
