# PRD 质量检查清单

> 确保每个 PRD 都可交付，不只是可读。

---

## 核心检查项

### 问题定义
- [ ] 问题陈述明确：谁、什么、为什么现在
- [ ] 不解决的后果有量化说明
- [ ] 从用户视角描述，非内部视角

### 目标和指标
- [ ] 主要目标单一、无歧义
- [ ] 每个指标有明确计算方式和数据来源
- [ ] 有基线值和目标值
- [ ] 有时间框架
- [ ] 包含护栏指标（防止负面影响）

### 解决方案
- [ ] 聚焦"做什么"而非"怎么做"
- [ ] 每个功能说明用户价值
- [ ] 与现有方案有差异化说明

### 需求规格（High-fi）
- [ ] 使用标准用户故事格式（As a... I want to... So that...）
- [ ] 每个故事有 ≥ 1 个 Given-When-Then 验收标准
- [ ] 考虑主要边缘情况

### 范围管理
- [ ] In Scope 明确
- [ ] Out Scope 明确（含排除原因）
- [ ] 延后内容有记录

### 风险和诚实
- [ ] 残酷风险区：牺牲了什么
- [ ] 残酷风险区：最大失败风险
- [ ] 残酷风险区：防御方案

---

## 按保真度分级

### 低保真（讨论稿）

必须：TL;DR + 问题陈述 + 成功指标 + 解决方案概述 + 范围 + 残酷风险区

可选：简化功能列表、主要依赖

### 高保真（工程就绪）

必须包含低保真全部 + 详细功能需求 + 技术考虑 + 依赖风险 + 时间线 + FAQ

---

## 常见问题速查

| 问题类型 | 反例 | 正例 |
|---------|------|------|
| 问题定义 | "用户需要更好的搜索" | "60% 用户在搜索 30 秒后放弃" |
| 成功指标 | "提升用户参与度" | "DAU 从 1 万到 1.5 万，30 天内" |
| 需求描述 | "实现个性化推荐" | 用户故事 + Given-When-Then |
| 范围边界 | 只说 In Scope | In Scope + Out Scope 都明确 |

---

## 质量评分

| 等级 | 标准 | 行动 |
|------|------|------|
| A | 所有核心检查项通过 | 可评审 |
| B | 1-2 项次要未通过 | 修补后评审 |
| C | 3+ 项未通过 | 重大修订 |
| D | 问题定义不清晰 | 重新定义 |
