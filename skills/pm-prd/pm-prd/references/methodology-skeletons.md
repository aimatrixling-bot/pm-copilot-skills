# PRD 方法论推理骨架

> Expert 模式下，根据产品类型自动选择推理骨架。骨架决定 PRD 的切入点和展开逻辑。

---

## Amazon Working Backwards

**适用**：B2B 企业功能、需要客户价值论证的场景

**推理逻辑**：从终局倒推——先写客户会怎么描述这个功能的价值

**产出结构**：
1. **Press Release（新闻稿）** — 假设功能已上线，写给客户的公告
   - 标题和副标题
   - 问题是什么
   - 解决方案一句话
   - 领导引言
   - 如何开始使用
2. **FAQ（常见问题）** — 内部和外部 FAQ
   - 外部：客户会问什么
   - 内部：工程/设计/运营会问什么
3. **User Stories** — 从 FAQ 中衍生出具体需求

**为什么有效**：强迫 PM 先想清楚客户价值，再想功能细节。防止"因为能做所以做"。

## JTBD（Jobs To Be Done）

**适用**：B2C 消费端功能、用户行为驱动的场景

**推理逻辑**：用户不是在"使用产品"，是在"完成一个任务"

**产出结构**：
1. **Job Statements** — 用户要完成的核心任务
   - 格式："当 {情境} 时，我想要 {动机}，以便 {期望结果}"
   - 每个任务对应一个 Job
2. **Current Pain** — 用户现在怎么完成这个任务，痛点是什么
3. **Solution Mapping** — 功能列表是 Job 的解决方案
   - 每个功能必须映射到一个 Job
   - 不能映射到任何 Job 的功能是"功能堆砌"
4. **PRD 展开** — 从 Job + Solution 出发写 PRD

**为什么有效**：防止功能堆砌，确保每个功能都有明确的用户任务支撑。

## Lean Canvas

**适用**：平台/基础设施、新业务线、商业模式不确定的场景

**推理逻辑**：先验证商业逻辑，再写功能需求

**产出结构**（9 格画布）：
1. **问题** — 前 3 个需要解决的问题
2. **客户细分** — 目标用户和早期采用者
3. **独特价值主张** — 一句话说明为什么不同且值得关注
4. **解决方案** — 前 3 个功能（对应前 3 个问题）
5. **渠道** — 如何触达用户
6. **收入来源** — 怎么赚钱
7. **成本结构** — 固定和可变成本
8. **关键指标** — 衡量成功的关键数字
9. **不公平优势** — 竞品无法轻易复制的优势

画布填充完成后再展开为完整 PRD。

**为什么有效**：确保功能需求建立在已验证的商业逻辑之上。

## 假设驱动

**适用**：AI 原生功能、技术不确定性高的场景

**推理逻辑**：先列出"如果 X 成立，功能才有价值"的假设，再写需求

**产出结构**：
1. **假设列表** — 列出所有关键假设
   - 格式："如果 {条件}，那么 {预期结果}"
   - 优先级排序：最危险的假设（推翻后影响最大的）排最前
2. **验证计划** — 每个假设的验证方法
   - 可用数据验证：写明数据来源和分析方法
   - 需要实验验证：写明实验设计和成功标准
3. **功能-假设映射** — 每个功能关联一个假设
   - 假设被推翻 → 对应功能需要重新评估
4. **PRD 展开** — 基于通过验证的假设写 PRD

**为什么有效**：AI 功能的价值高度依赖模型能力，假设驱动防止过度承诺。

---

## 标准 PRD

**适用**：不确定用哪种、或功能相对常规的场景

**推理逻辑**：使用模板按部就班填充

**产出结构**：直接按 `references/prd-template.md` 中的完整模板填充。

---

## 选择指南

| 场景 | 推荐骨架 | 判断信号 |
|------|---------|---------|
| 客户明确，需要论证价值 | Working Backwards | 用户提到"客户"、"合同"、"销售" |
| 用户行为复杂，需要理解动机 | JTBD | 用户提到"用户想要"、"行为"、"场景" |
| 商业模式不确定 | Lean Canvas | 用户提到"商业模式"、"收入"、"市场" |
| 技术不确定性高 | 假设驱动 | 用户提到"AI"、"模型"、"如果能做到" |
| 常规功能迭代 | 标准 PRD | 用户直接说"加个 XX 功能" |
