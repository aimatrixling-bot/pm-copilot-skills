# Backlog 管理方法

> 需求生命周期管理的实用指南。

---

## 1. 需求状态流转

```
提议中 → 分析中 → 已排期 → 开发中 → 已发布 → 已归档
           ↓          ↓
        暂缓       已取消
```

| 状态 | 进入条件 | 退出条件 | 典型停留 |
|------|----------|----------|----------|
| **提议中** | 需求被提出 | 开始分析 | 不定 |
| **分析中** | 需求澄清中 | 技术可行、范围清晰 | 1-2 周 |
| **已排期** | 优先级确定、资源分配 | 进入 Sprint | 1-4 周 |
| **开发中** | 进入当前 Sprint | 开发完成、通过 QA | 1-2 周 |
| **已发布** | 功能上线验收 | 归档期结束 | 持续 |
| **已归档** | 发布后复盘完成 | — | 永久 |
| **已取消** | 决定不做 | — | 永久 |
| **暂缓** | 暂时搁置 | 重新激活或取消 | 不定 |

## 2. 需求拆分原则

### INVEST 原则

| 字母 | 含义 | 检查 |
|------|------|------|
| I | Independent（独立） | 可独立交付，不依赖其他任务 |
| N | Negotiable（可协商） | 实现方式可讨论 |
| V | Valuable（有价值） | 对用户/业务有价值 |
| E | Estimable（可估算） | 团队能估算工作量 |
| S | Small（小） | 每个 ≤ 2 天工作量 |
| T | Testable（可测试） | 有明确的完成标准 |

### 拆分技巧

| 技巧 | 适用场景 |
|------|----------|
| 按用户操作步骤拆 | 工作流类功能 |
| 按数据 CRUD 拆 | 数据管理类功能 |
| 按规则/条件拆 | 业务规则复杂的功能 |
| 按界面拆 | 多页面功能 |
| Happy path 先行 | 复杂功能先做主流程 |

## 3. Sprint 容量估算

### 估算方法

| 方法 | 适用团队 | 精度 |
|------|----------|------|
| 故事点 | 成熟敏捷团队 | 中 |
| 人天 | 新团队 | 中低 |
| T恤尺码 | 快速粗估 | 低 |

### 容量计算

```
团队总人数 × Sprint 天数 × 有效工作时间比(≈70%)
= 理论容量

理论容量 - 假期 - 会议 - On-call
= 实际容量

实际容量 × 80%（只承诺 80%）
= Sprint 承诺容量
```

## 4. 优先级排序方法

### 快速排序（MoSCoW）

| 类别 | 含义 | 占比建议 |
|------|------|----------|
| Must have | 不做就不发布 | ~40% |
| Should have | 重要但不阻塞 | ~30% |
| Could have | 锦上添花 | ~20% |
| Won't have | 明确排除 | ~10% |

### RICE 排序（更精确）

使用 `pm-rice` Skill 进行量化评分。

## 5. 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 任务太大（>2天） | 无法跟踪进度 | 拆分为 ≤2 天的任务 |
| 无依赖标注 | 任务互相阻塞 | 列出所有跨任务依赖 |
| Sprint 过度承诺 | 总是完不成 | 只承诺 80% 容量 |
| 不更新状态 | 看不到真实进度 | 每日更新任务状态 |
| 需求来源不清 | 无法追溯决策 | 每个需求标注来源 |
