# 需求评审质量检查清单

> 本清单用于评估 `/critique` 产出的质量

---

## 核心检查项 (🔴 必需)

### 内容理解

- [ ] **理解准确**: 对需求的理解准确无误
- [ ] **背景明确**: 需求背景和目标清晰
- [ ] **范围清晰**: 评审范围明确

### 评审执行

- [ ] **视角恰当**: 选择了合适的评审视角
- [ ] **覆盖全面**: 所选视角的评审全面
- [ ] **问题具体**: 问题描述具体明确
- [ ] **影响分析**: 每个问题有影响分析

### 建议质量

- [ ] **建议可行**: 改进建议具体可行
- [ ] **优先级清晰**: 问题按严重程度分类
- [ ] **责任明确**: 有明确的负责人和时间

### 报告质量

- [ ] **结构清晰**: 报告结构清晰易读
- [ ] **平衡性**: 平衡问题和正面发现
- [ ] **可执行**: 报告结论可执行

---

## 常见问题检测

| 问题 | 检测方法 | 修改建议 |
|------|----------|----------|
| 理解偏差 | 与需求方确认 | 评审前确认理解 |
| 只批评无建议 | 只有问题列表 | 补充改进建议 |
| 过于笼统 | 问题描述模糊 | 具体化问题和建议 |
| 无优先级 | 不知道先修哪个 | 按严重程度分类 |
| 视角单一 | 遗漏其他问题 | 多视角评审 |

## 好的评审特征

### 准确理解

评审前确认理解：

```
**需求理解**:
- 背景: [背景]
- 目标: [目标]
- 用户: [用户]

请确认以上理解是否正确。
```

### 具体问题

问题描述具体且有上下文：

❌ 流程有问题
✅ 用户在步骤 3 需要返回步骤 1，但缺少返回按钮，导致流程中断

### 平衡性

平衡问题和正面：

**正面发现**:
- 需求定位清晰
- 目标用户明确

**问题发现**:
- [问题列表]

### 可行建议

建议具体可行：

❌ 改进流程
✅ 建议: 在步骤 3 添加"返回"按钮，链接到步骤 1，预计工作量 0.5 天

### 优先级清晰

按严重程度排序：

| 优先级 | 问题数 | 处理建议 |
|--------|--------|----------|
| P0 严重 | N | 必须解决，阻塞发布 |
| P1 中等 | N | 应该解决，影响质量 |
| P2 轻微 | N | 可以改进，优化体验 |
