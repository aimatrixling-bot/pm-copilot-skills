# 功能开发全周期指南

> 从需求到上线的六阶段执行手册。

---

## 1. 阶段转换标准

### Phase 1 → Phase 2（优先级 → PRD）

| 必须完成 | 检查方式 |
|----------|----------|
| RICE 四维度无缺失 | Reach/Impact/Confidence/Effort 均有数值 |
| P0 功能已确认 | 有明确的 P0 列表 |
| 排序结果用户已确认 | 用户认可排序 |

### Phase 2 → Phase 3（PRD → 技术规格）

| 必须完成 | 检查方式 |
|----------|----------|
| PRD Iron Law 通过 | Why/验收标准/成功指标齐全 |
| Feature Scope 冻结 | 功能范围不再变化 |
| 验收标准可测试 | 每个 AC 有 Given-When-Then |

### Phase 3 → Phase 4（技术规格 → 工程需求）

| 必须完成 | 检查方式 |
|----------|----------|
| 技术方案有评审 | 至少 1 位技术 lead 确认 |
| 架构决策有依据 | 每个 ADR 有原因 |
| 风险已列出 | ≥ 3 个技术风险 |

### Phase 4 → Phase 5（工程需求 → 测试）

| 必须完成 | 检查方式 |
|----------|----------|
| 任务已拆分 | 每个任务 ≤ 2 天 |
| 验收标准完整 | 每个任务有 AC |
| 依赖已标注 | 跨团队依赖已沟通 |

### Phase 5 → Phase 6（测试 → 发布）

| 必须完成 | 检查方式 |
|----------|----------|
| 测试计划覆盖全部 US | 每个 User Story 有测试用例 |
| P0 场景 100% 通过 | 所有关键路径测试通过 |
| Bug 分级处理 | P0 全修、P1 评估、P2 记录 |

---

## 2. 常见卡点

| 卡点 | 原因 | 解决方案 |
|------|------|----------|
| Phase 1 卡住 | 功能太多无法排序 | 先筛选 ≤ 10 个功能再排序 |
| Phase 2 卡住 | PRD 写不完 | 用 Quick 模式先出 v0.1 |
| Phase 3 卡住 | 技术方案有争议 | 列出 2-3 个备选用 pm-decision 决策 |
| Phase 4 卡住 | 估时不准确 | 标注 [待确认]，让开发参与估算 |
| Phase 5 卡住 | 测试用例太多 | P0 必须 100%、P1 覆盖主流程、P2 抽检 |
| Phase 6 卡住 | 发布前发现 P0 Bug | 回到 Phase 5 修复，不带着 P0 上线 |

---

## 3. Quick vs Full 模式

| 模式 | 适用场景 | 产出深度 |
|------|----------|----------|
| Quick | 时间紧、需要快速验证 | 每阶段 1-2 页核心内容 |
| Full | 正式项目、需要完整文档 | 每阶段完整产出 |

### Quick 模式省略项

- Phase 1：简化为 Top-3 排序（不做完整 RICE）
- Phase 2：1 页 PRD（Why + 验收标准 + 成功指标）
- Phase 3：技术决策记录（不做完整技术规格）
- Phase 4：任务列表（不做完整工程需求单）
- Phase 5：P0 测试用例（不做完整测试计划）
- Phase 6：发布检查清单（不做完整发布计划）
