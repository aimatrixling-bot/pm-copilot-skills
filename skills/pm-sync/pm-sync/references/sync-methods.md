# 项目同步方法

> 状态报告的生成策略和变化识别方法。

---

## 1. 文件扫描策略

### 扫描范围

| 优先扫描 | 文件模式 |
|----------|----------|
| PM 文档 | `*.md`, `PRD*`, `docs/**` |
| 项目配置 | `package.json`, `CLAUDE.md`, `*.yaml` |
| Registry | `*-registry.md` |
| 测试 | `**/*.test.*`, `**/*.spec.*` |

### 时间过滤

| 报告类型 | 扫描范围 |
|----------|----------|
| daily | 最近 24 小时修改的文件 |
| weekly | 最近 7 天修改的文件 |
| milestone | 自上次 milestone 以来的所有变化 |

---

## 2. 变化识别方法

### 什么是"有意义的"变化

| 变化类型 | 示例 | 是否报告 |
|----------|------|----------|
| 新增文档 | 新建 PRD | 是 |
| 内容变更 | 修改验收标准 | 是 |
| 状态变化 | "进行中" → "已完成" | 是 |
| 新增风险 | 识别新依赖 | 是 |
| 格式调整 | 修改标题层级 | 否 |
| 拼写修正 | 修正错别字 | 否 |
| 重命名 | 文件改名 | 仅标注 |

### 变化提炼原则

1. **只说 what + why**，不说 how（细节链接到原文件）
2. **量化变化**："新增 3 个 User Story" > "更新了 PRD"
3. **标注来源**：每个变化标注 `(来源: path/to/file)`

## 3. 三种报告格式差异

### Daily（日报）

```markdown
## 项目日报 — [日期]

### 今日变化
- [变化 1] (来源: file.md)
- [变化 2] (来源: file.md)

### 阻塞项
- [阻塞项]
```

**特点**: 极简，≤ 10 行。只列变化和阻塞。

### Weekly（周报）

```markdown
## 项目周报 — [日期]

### 本周进展
- [进展 1] (来源: file.md)
- [进展 2] (来源: file.md)

### 待处理
- [待确认/待决策事项]

### 风险
- [新识别的风险]

### 下周计划
- [计划事项]
```

**特点**: 完整结构，≤ 1 页。进展 + 待处理 + 风险 + 计划。

### Milestone（里程碑报告）

```markdown
## 里程碑报告 — [里程碑名]

### 里程碑目标
- [目标 1]
- [目标 2]

### 达成情况
| 目标 | 状态 | 说明 |
|------|------|------|
| [目标 1] | ✅/⚠️/❌ | [说明] |

### 关键决策
- [决策 1]: [原因] (来源: file.md)

### 遗留问题
- [问题 1]: [影响] - [负责人]

### 下一里程碑
- [里程碑名]: [目标]
```

**特点**: 目标导向，评估达成率。

## 4. 常见错误

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 扫描所有文件 | 信息过载 | 只扫描 PM 相关文件 |
| 不标注来源 | 无法验证 | 每个变化标注来源文件 |
| 格式变化也报告 | 噪音太多 | 只报告内容变化 |
| 报告太长 | 没人看 | 日报 ≤ 10 行，周报 ≤ 1 页 |
| 不识别风险 | 问题积累 | 主动从文件中识别风险信号 |
